# fix: Scroll container when FOCUS or CURSOR moves beyond visible area

## Problem

When navigating with FOCUS (Nav) or CURSOR (TokenCursor) inside a scrollable container (e.g. `overflow-y: scroll`), the container doesn't scroll to keep the focused/cursored element visible. The element moves off-screen and the user loses track of where they are.

Reproducible using the "Scrollable container" example in `apps/jsed-demo/static/test_doc.html`.

## Goal

When FOCUS or CURSOR moves to an element that is outside the visible area of its scroll container, scroll the container to bring the element into view.

## How to do this

1. Read `packages/jsed/CLAUDE.md` and the `/jsed` skill to understand the codebase structure. Read `packages/jsed/docs/vocabulary.md` to understand FOCUS, CURSOR, TOKEN, and LINE.

2. **Understand the current flow:**
   - FOCUS movement is handled by `Nav` (`packages/jsed/src/Nav.ts`). Look at the methods that change focus — `focusDown`, `focusUp`, `focusPrev`, `focusNext`, etc. Identify where the focused element changes.
   - CURSOR movement is handled by `TokenCursor` (`packages/jsed/src/TokenCursor.ts`). Look at `moveNext`, `movePrevious`, and `setTokenInternal` — these change which element the cursor is on.
   - There is already a `scrollIntoView` call somewhere in `InputManager.ts` — check what it does and whether it can be reused or extended.

3. **Choose where to add scroll behaviour:**
   - The simplest approach is `element.scrollIntoView({ block: 'nearest', inline: 'nearest' })` after the focus/cursor element changes. `block: 'nearest'` avoids jarring jumps — it only scrolls if the element is outside the viewport.
   - For FOCUS: the right hook is likely after `#setFocus` in Nav, or wherever the focus CSS class is applied.
   - For CURSOR: the right hook is likely `setTokenInternal` in TokenCursor, or the `onTokenChange` callback.
   - Consider whether this should live in the existing classes or in the `EditManager` orchestrator that wires them together.

4. **Be careful about:**
   - Don't scroll on initial load / first focus — only on navigation moves
   - Don't scroll when the element is already visible — `block: 'nearest'` handles this
   - The `ElementIndicator` already has scroll/scrollend handlers that hide/show the badge during scroll — make sure your scroll-into-view doesn't fight with this. The indicator hides on scroll and re-shows on scrollend, which should work fine with programmatic scrolls too.

5. **Test manually** using `apps/jsed-demo/`:
   - `task dev` to start the demo
   - Use the "Scrollable container" example in test_doc.html
   - Navigate with FOCUS (arrow keys) past the visible area — container should scroll
   - Navigate with CURSOR (within a tokenized line) past the visible area — container should scroll
   - Verify the ElementIndicator badge still appears correctly after scrolling

6. **Write a unit test** if there's a clean way to verify scroll was called (e.g. via the nullables pattern — Nav and TokenCursor both have `createNull`). If not, manual testing is sufficient for this fix.
