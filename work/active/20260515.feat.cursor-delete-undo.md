# feat: Cursor delete undo

Add the first undo mechanism for jsed by targeting text changes made through `Cursor`, starting with `CursorTextOps.delete()` in `packages/jsed/src/lib/cursor/CursorTextOps.ts`.

The first version should support a single linear undo history shared by human and agent edits. Human and agent operations are synchronized so only one semantic editor operation runs at a time. Undo is global and strictly ordered: it is not possible to undo only the agent's changes or only the human's changes.

## Design direction

Use semantic operation results as the source of truth for undo, not raw `MutationObserver` records.

`MutationObserver` can identify low-level DOM changes, including `target`, `removedNodes`, `addedNodes`, `previousSibling`, and `nextSibling`, but it records DOM mutations rather than jsed operations. `CursorTextOps.delete()` can remove the current `TOKEN`, remove separator whitespace, insert an `ANCHOR`, prune empty ancestors through `deleteEmptyTree`, and move the `CURSOR`. Those mutations belong to one semantic operation and should be represented as one result.

Prefer this shape:

```ts
const result = cursor.delete();

if (result.type === 'deleted-token') {
  history.record(result);
}
```

Lower-level helpers should return meaningful mutation facts where the caller already needs to reason about the consequence. Avoid creating parallel `fooTracked` helpers for every mutating function unless changing the existing return type would be too disruptive.

Examples:

```ts
const removal = token.remove(current);
const pruned = deleteEmptyTree(parent, root);
```

`CursorTextOps.delete()` composes those lower-level results into a semantic `CursorDeleteResult`.

## Proposed result shapes

Represent local insertion sites with live DOM references:

```ts
type DomInsertionSite = {
  parent: Node;
  previousSibling: Node | null;
  nextSibling: Node | null;
};
```

Represent token removal with enough information to restore the deleted `TOKEN` and any separator whitespace:

```ts
type TokenRemoval = {
  token: HTMLElement;
  tokenSite: DomInsertionSite;
  separatorBefore: Text | null;
  separatorBeforeSite: DomInsertionSite | null;
  separatorAfter: Text | null;
  separatorAfterSite: DomInsertionSite | null;
  previousVisibleSibling: HTMLElement | null;
  nextVisibleSibling: HTMLElement | null;
};
```

Represent cursor deletion as a semantic operation result:

```ts
type CursorDeleteResult =
  | { type: 'noop'; reason: 'not-on-token' | 'anchor' }
  | {
      type: 'deleted-token';
      deletionType: 'charDeletion' | 'tokenDeletion';
      removal: TokenRemoval;
      insertedAnchor: HTMLElement | null;
      removedEmptyAncestors: RemovedElement[];
      cursorBefore: HTMLElement;
      cursorAfter: HTMLElement;
      inputCursorPosition: UserInputOpts['inputCursorPosition'];
    };
```

If `deleteEmptyTree()` prunes empty ancestors, record each removed element and its own insertion site so undo can restore ancestors before restoring the deleted `TOKEN`.

```ts
type RemovedElement = {
  element: HTMLElement;
  site: DomInsertionSite;
};
```

## Undo assumptions

The initial undo stack can assume strict LIFO ordering.

If a deletion records `previousSibling` and `nextSibling`, and those siblings are deleted by later operations, that is acceptable because those later operations must be undone first. By the time the earlier deletion is undone, the sibling references or parent structure should have been restored by newer undo entries.

Restore using a helper that prefers attached reference siblings:

```ts
function restoreAtSite(site: DomInsertionSite, node: Node): void {
  if (site.nextSibling?.parentNode === site.parent) {
    site.parent.insertBefore(node, site.nextSibling);
    return;
  }

  if (site.previousSibling?.parentNode === site.parent) {
    site.parent.insertBefore(node, site.previousSibling.nextSibling);
    return;
  }

  site.parent.appendChild(node);
}
```

Undo for a delete should restore removed ancestors first, then separator nodes and the deleted `TOKEN`, then place the `CURSOR` on the restored token or the appropriate semantic target.

## Tombstone decision

Do not add a tombstone marker in the first version.

A DOM tombstone marked as `IGNORABLE` could later provide a durable live insertion marker for a delete site. If introduced, it should be a first-class taxonomy term or editor-owned infrastructure node, skipped by `FOCUS`, `CURSOR`, `TOKENIZATION`, detokenization, emptiness checks, and serialization.

For now, use recorded insertion sites and live DOM references.

## Agent and collaboration direction

Record actor metadata on history entries, but keep undo global:

```ts
type HistoryEntry = {
  id: string;
  actor: 'human' | 'agent';
  label: string;
  timestamp: number;
  inverse: UndoCommand;
};
```

Agent operations may later need explicit transactions if one high-level agent action contains several jsed operations. For the first version, keep transactions synchronized and non-overlapping.

## Operational transform considerations

Leave the door open to remote operations and operational transform by distinguishing local undo acceleration from durable operation identity.

Live DOM references are acceptable for the first local LIFO undo implementation, but they are not enough for OT. Future remote-capable operations should be serializable, revision-aware, and address document targets without relying only on node identity.

Likely future shape:

```ts
type OperationEnvelope = {
  opId: string;
  actorId: string;
  baseRevision: string;
  parents: string[];
  op: JsedOperation;
};
```

Future `JsedOperation` values should be vocabulary-level operations such as `DELETE_TOKEN`, `INSERT_TOKEN`, `REPLACE_TOKEN`, `SPLIT_BY_TOKEN`, and `JOIN`. Durable addressing will probably require stable ids on editable structural nodes and `TOKEN` elements, or a hybrid of stable ids plus sibling/context fallback.
