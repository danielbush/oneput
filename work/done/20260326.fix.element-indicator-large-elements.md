# fix: ElementIndicator positioning for large elements

## Problem

When a FOCUSABLE element is taller than the viewport, the ElementIndicator badge can appear at the **bottom** of the element — potentially off-screen — even when the element's top is visible in the viewport. The indicator should always be visible to the user.

## Investigation

Before changing behaviour, investigate the current logic in `src/ElementIndicator.ts`, specifically `#makeIndicator`:

1. **Why does `positionBelow` trigger?** The current logic checks `rect.top - 5 < indicatorHeight` — if there's not enough space above the element's top edge to fit the badge, it positions below (`rect.bottom + 5`). For a large element, `rect.bottom` may be far below the viewport. This logic was likely written for elements near the top of the viewport where flipping below makes sense (small element, top clipped). It doesn't account for elements that are tall.
2. **When is below-positioning correct?** Only when the element is small and its top is near/above the viewport top. If the element's top IS within the viewport, the badge should be near the top — not below.
3. **What about the IntersectionObserver?** It uses `threshold: 0`, so any pixel of the element being visible counts as "intersecting". For a large element, the element stays "visible" even as the top scrolls past. This is correct for visibility but the positioning logic doesn't account for the top being off-screen.

## Expected behaviour

### Case 1: Element top is within the viewport
- Position the indicator near the **top-right corner** of the element (inside the element bounds)
- If the element is narrow and right-aligning would clip the left edge, apply left-align logic (already exists)

### Case 2: Element top has scrolled above the viewport
- Position the indicator at the **top of the viewport** (pinned), horizontally aligned with the element
- The indicator stays visible as the user scrolls through the large element

### Case 3: Small element fully in viewport (existing behaviour)
- Keep existing above-element positioning
- Only flip below if the element's top is truly clipped by the viewport top AND the element is small enough that below-positioning is still on-screen

## Instructions

### Files to modify
- `packages/jsed/src/ElementIndicator.ts` — the `#makeIndicator` method contains all positioning logic

### Constraints
- Follow the nullables pattern already established in this file (the class has `create()` / `createNull()`)
- Do not change the public API (`showIndicator`, `updateFocus`, `destroy`)
- The indicator uses `position: fixed` so coordinates are relative to the viewport (`getBoundingClientRect` values)
- Keep the `JSED_IGNORE_CLASS` on the indicator element
- Preserve left-align logic for narrow elements

### Approach
1. In `#makeIndicator`, after getting `rect`, determine which case applies:
   - `rect.top >= 0 && rect.bottom <= viewportHeight` → small element, use existing logic
   - `rect.top >= 0 && rect.bottom > viewportHeight` → large element, top visible → anchor inside top-right corner
   - `rect.top < 0 && rect.bottom > 0` → top scrolled past → pin to viewport top, aligned with element
2. For the "pinned to viewport top" case, the indicator should sit at `top: 5px` (or similar small offset)
3. The `#scrollEndHandler` already re-runs `#addIndicator` after scroll, so the pinned position updates automatically
4. Write tests using `createNull()` — but note that `#makeIndicator` calls `getBoundingClientRect` and `offsetHeight`/`offsetWidth`, so you may need to extend the null infrastructure to support configurable rects. Discuss this in the Changes section if the test infrastructure needs non-trivial changes.
