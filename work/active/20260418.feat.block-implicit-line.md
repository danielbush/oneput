# feat: block-based IMPLICIT_LINE

## Motivation

The old IMPLICIT_LINE was a cursor-transparent `<span>` used both to wrap trailing text after a LINE *and* to wrap bare text runs interleaved with nested blocks. The cursor-transparent semantics were load-bearing for within-LINE descent (e.g. `<div>aaa<div>nested</div>bbb</div>`: outer `<div>` was the LINE, implicit-lines around `aaa`/`bbb` were descended through to reach the tokens). This forced workarounds in two places so cross-LINE navigation could step *into* IMPLICIT_LINE's:

- `Tokenizer.tokenizeLineAt` special-cased `isImplicitLine(el) ? el : findLineCandidateAt(el).line`.
- `sibwalk.findNextLineCandidate`/`findPreviousLineCandidate` widened their visit predicate to include non-whitespace text nodes, then lifted them to `parentElement`.

Both are symptoms of IMPLICIT_LINE being "a LINE for tokenization, not a LINE for navigation".

## Design

**Rule:** when a LINE contains a block child, partition its non-block children into runs separated by blocks. Each non-empty run (text + INLINE_FLOW's) is wrapped in a block-IMPLICIT_LINE — `<div class="jsed-implicit-line">` — which is a true LINE. A LINE with only inline content gets no IMPLICIT_LINE's.

Consequences:

- IMPLICIT_LINE is no longer cursor-transparent.
- `isLine` returns true for IMPLICIT_LINE.
- Both workarounds above disappear.
- `<div>aaa<div>nested</div>bbb</div>` becomes three LINEs, not one: `implicit-line(aaa)`, `nested-div`, `implicit-line(bbb)`. Cleaner structural model.

## Status (2026-04-18)

Algorithm landed in `packages/jsed/src/lib/implicitLine.ts`. Downstream cleanup applied: `taxonomy.ts` (IMPLICIT_LINE is now a LINE and not cursor-transparent), `Tokenizer.ts` (special case removed), `sibwalk.ts` (widening reverted). Typecheck clean.

36 unit tests fail because they encode the old "outer block + nested block = one LINE with a descended-into block" model. Failures cluster into four groups; each becomes an action item below.

## Guidance for fixing failing tests

- Most of these tests only care about *what the CURSOR visits in order* — they don't need the machinery to run real tokenization. Where possible, replace tokenize-then-walk fixtures with hand-built DOMs using the `t()` / `s()` helpers in `src/test/util.ts` (pre-tokenized text runs + whitespace nodes).
- Several of these assertions are really about editor-level behaviour (navigation, whitespace-around-OPAQUE_BLOCK). Per `packages/jsed/CLAUDE.md`, tests should live at the `EditManager` level where possible. Use this refactor as the opportunity to move them up instead of duplicating low-level sibwalk/tokenize coverage.
- Keep low-level `sibwalk`/`tokenize` tests only where they cover a specific primitive not observable at the EditManager level.

## Changes

- **2026-04-18** — created ticket. Algorithm + taxonomy/Tokenizer/sibwalk cleanup landed; four groups of test failures queued as action items.
- **2026-04-18** — exploratory surfaced broken cross-LINE navigation. Root cause: `sibwalk.findNextLineCandidate`/`findPreviousLineCandidate` were reverted using the wrong visit predicate (`isCandidateLineSibling`, which only matches TOKEN/ISLAND/text-with-content) instead of the original `isLineSibling` (any non-transparent FOCUSABLE). The walker skipped straight past sibling LINEs because a `<p>` / implicit-line-div is neither a TOKEN nor a text node. Fixed both to use `isLineSibling`. Test-failure count 36 → 28.
- **2026-04-18** — FOCUS was landing on IGNORABLE elements in the demo. Cause: `tagImplicitLines` was pulling IGNORABLEs into IMPLICIT_LINE wrappers (which are LINEs, therefore a focusable context). Fixed: IGNORABLEs are now pass-over inside `tagImplicitLines` — never absorbed, never boundaries, walker doesn't descend into them.
- **2026-04-18** — floats are now pass-over in `tagImplicitLines` (same treatment as IGNORABLEs inside the algorithm, but floats remain fully FOCUSABLE LINEs in the broader taxonomy). "Out of flow" means surrounding text runs partition as if the float weren't there. Before: `<div><p/><float/>trailing</div>` partitioned at both `<p>` and the float, and `<p>aaa<float/>bbb</p>` got torn into three LINEs by the float. After: float stays where it is, doesn't fragment runs. New test `floats are pass-over` in `implicitLine.test.ts`.
