# discussion: FOCUSABLE display classification is inconsistent

The taxonomy in `docs/vocabulary.md` defines a clean hierarchy (INLINE_FLOW, NON_INLINE_FLOW, OPAQUE_BLOCK, TRANSPARENT_BLOCK, ISLAND) but the implementation uses at least three different strategies to decide where an element falls, and they don't always agree on what counts as "inline-like."

## The three strategies

### 1. `isInlineFlow()` in traversal.ts — narrow definition

```typescript
if (styles.display === 'inline') return true;
if (styles.display === 'inline flow') return true;
return false;
```

Only `inline` and `inline flow` count. An `inline-block`, `inline-flex`, or `inline-grid` element is **not** INLINE_FLOW — it falls through to `isLine()` and becomes an OPAQUE_BLOCK by default. Float check is also here (floated elements are never INLINE_FLOW).

### 2. `tagImplicitLines()` in token.ts — broad definition

```typescript
const display = window.getComputedStyle(prev as HTMLElement).display;
if (!display.startsWith('inline')) {
  // wrap in IMPLICIT_LINE
}
```

Anything starting with `inline` (including `inline-block`, `inline-flex`, `inline-grid`) is treated as "doesn't cause a visual line break" and skipped. This is a **wider** net than `isInlineFlow()`.

### 3. `isLine()` in traversal.ts — negative definition

```typescript
return isFocusable(el) && !isToken(el) && !isInlineFlow(el) && !isIsland(el);
```

Everything that isn't INLINE_FLOW, TOKEN, or ISLAND becomes a LINE. Since `isInlineFlow()` is narrow, an `inline-block` becomes a LINE, then an OPAQUE_BLOCK (because it won't have `jsed-cursor-transparent`). But `tagImplicitLines` already decided it *doesn't* cause a line break.

## Where this gets confusing

### inline-block / inline-flex elements

- `isInlineFlow()` says: **not inline** → falls into LINE → OPAQUE_BLOCK
- `tagImplicitLines()` says: **inline-ish** → no IMPLICIT_LINE needed
- Vocabulary says: OPAQUE_BLOCK includes "block-like things like inline-blocks"
- Result: an inline-block is an OPAQUE_BLOCK that the CURSOR visits but doesn't descend — yet visually it sits on the same line as surrounding text and doesn't trigger IMPLICIT_LINE wrapping. Is this correct? It means the cursor will land on it as a LINE_SIBLING, which might actually be the right UX, but the reasoning path is hard to follow.

### Floats

- `isInlineFlow()` explicitly excludes floated elements (even if `display: inline`)
- `tagImplicitLines()` doesn't check float at all — a floated element with `display: block` won't start with `inline` so it triggers IMPLICIT_LINE wrapping. But a floated `display: inline` element also won't match `isInlineFlow()` so it becomes a LINE.
- Question: should `tagImplicitLines` also consider float? A floated element is visually pulled out of flow.

### Direct getComputedStyle vs is* wrappers

- `tagImplicitLines` reads `getComputedStyle` directly and uses `startsWith('inline')`
- `isInlineFlow()` reads `getComputedStyle` and checks exact values
- If the definition of "inline-like" ever needs to change (e.g. to handle a new display value), you'd need to find and update both — and they might drift further apart.

## Proposed primitives and taxonomy table

The entire taxonomy can be derived from three independent predicates plus one class flag:

- `isFocusable` — can the user navigate to this element?
- `isIsland` — externally managed content (katex, `<img>`, form controls)?
- `isInlineFlow` — narrow display predicate: CSS `inline` or `inline flow` only. Drives the taxonomy table.
- `isCursorTransparent` — does the cursor descend through this element without visiting it?

There is also a second display predicate for IMPLICIT_LINE triggering:

- `isInline` — broad display predicate: any `inline*` display value (`startsWith('inline')`). Includes `inline-block`, `inline-flex`, `inline-grid`, etc. Used by `tagImplicitLines` to decide whether an element causes a visual line break.

Both are pure display predicates — no focusable/island/class checks. `isInlineFlow` implies `isInline` but not vice versa (an `inline-block` is `isInline` but not `isInlineFlow`).

`isInlineFlow` and `isIsland` are independent — an `<img>` with `display: inline` is both.

```
┌─────────────┬───────────┬───────────────┬─────────────────────┬───────────────────┬────────────────────┐
│             │           │               │ isCursorTransparent │ Derived label     │ Cursor behavior    │
├─────────────┼───────────┼───────────────┼─────────────────────┼───────────────────┼────────────────────┤
│             │           │ isInlineFlow  │ no                  │ INLINE_ISLAND     │ CURSOR_OPAQUE      │
│             │ isIsland  ├───────────────┼─────────────────────┼───────────────────┼────────────────────┤
│             │           │ !isInlineFlow │ no                  │ NON_INLINE_ISLAND │ CURSOR_OPAQUE      │
│ isFocusable ├───────────┼───────────────┼─────────────────────┼───────────────────┼────────────────────┤
│             │           │ isInlineFlow  │ yes                 │ INLINE_FLOW       │ CURSOR_TRANSPARENT │
│             │           ├───────────────┼─────────────────────┼───────────────────┼────────────────────┤
│             │ !isIsland │               │ yes (class)         │ TRANSPARENT_BLOCK │ CURSOR_TRANSPARENT │
│             │           │ !isInlineFlow ├─────────────────────┼───────────────────┼────────────────────┤
│             │           │               │ no  (default)       │ OPAQUE_BLOCK      │ CURSOR_OPAQUE      │
└─────────────┴───────────┴───────────────┴─────────────────────┴───────────────────┴────────────────────┘
```

`isCursorTransparent` is fully determined by the other predicates except in the `!isIsland && !isInlineFlow` case, where the `jsed-cursor-transparent` class is the tiebreaker. Everything else — OPAQUE_BLOCK, TRANSPARENT_BLOCK, CURSOR_OPAQUE, LINE, isLineSibling — is derived:

- `isCursorOpaque` = `!isCursorTransparent`
- `isLine` = `!isIsland && !isInlineFlow`
- `isLineSibling` = `isToken || isCursorOpaque`

### Where "triggers IMPLICIT_LINE" breaks the table

Adding a column for "triggers IMPLICIT_LINE" (does text after this element get wrapped?) exposes the mismatch. For ISLAND and INLINE_FLOW rows it's clean — ISLANDs with non-inline display trigger it, INLINEs don't. But the `!isIsland && !isInlineFlow` rows would need to split:

```
OPAQUE_BLOCK (block, flex, grid...)          │ triggers IMPLICIT_LINE: yes
OPAQUE_BLOCK (inline-block, inline-flex...)  │ triggers IMPLICIT_LINE: no
```

This split doesn't exist in the taxonomy today. `tagImplicitLines` uses `startsWith('inline')` (broad) while `isInlineFlow` is narrow (`inline` and `inline flow` only). An `inline-block` is classified as OPAQUE_BLOCK but doesn't trigger IMPLICIT_LINE wrapping — because the two checks draw the boundary differently.

## Possible work

### 1. Replace `isInline` with `isInlineFlow` as a pure display predicate

The current `isInline` is tangled — it checks `isFocusable`, `isIsland`, implicit-line class, float, *and* display. `isInlineFlow` should be a pure display predicate: does this element have inline-flow display? No island detection, no focusable check, no class checks. Just the CSS display model.

This means:
- `isInlineFlow` and `isIsland` become truly independent primitives
- Code that currently calls `isInline` gets replaced with the appropriate combination of primitives (e.g. `isInlineFlow(el) && !isIsland(el)` where both conditions matter)
- The existing derived labels (INLINE_FLOW, OPAQUE_BLOCK, etc.) still work — they're just expressed as combinations in the table above rather than baked into a single function

### 2. Remove or redefine other `is*` functions

With `isFocusable`, `isIsland`, `isInlineFlow`, and `isCursorTransparent` as primitives, explore which existing functions survive, get redefined, or disappear:

- `isInline` → replaced by `isInlineFlow` (pure) + combinations where needed
- `isLine` → derived: `!isIsland && !isInlineFlow` (given focusable, not token)
- `isCursorBoundary` → derived: `!isCursorTransparent` (i.e. `isCursorOpaque`)
- `isTransparentBlock` → derived: `isCursorTransparent && !isInlineFlow`
- `isLineSibling` → derived: `isToken || isCursorOpaque`

The goal is fewer named predicates, each doing one thing, with the taxonomy falling out of combinations rather than being encoded in bespoke functions.

### 3. IMPLICIT_LINE triggering

`tagImplicitLines` currently uses raw `getComputedStyle` + `startsWith('inline')`. Replace with `isInline` (the broad display predicate). Same semantics, but now it's a named predicate — no raw display checks outside the primitive layer.
