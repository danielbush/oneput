## Issues

The basic restoration helper records both surrounding siblings, because strict LIFO undo means later edits should have been undone before this plan is restored:

```ts
type InsertionSite = {
  parent: Node;
  previousSibling: Node | null;
  nextSibling: Node | null;
};

function restoreAtInsertionSite(site: InsertionSite, node: Node): void {
  if (site.nextSibling?.parentNode === site.parent) {
    site.parent.insertBefore(node, site.nextSibling);
    return;
  }

  if (site.previousSibling?.parentNode === site.parent) {
    site.parent.insertBefore(node, site.previousSibling.nextSibling);
    return;
  }

  site.parent.appendChild(node);
}
```

Assumption: separator whitespace remains rule-based for now. `RemoveTokenPlan` records candidate separators so execution and undo can apply spacing rules algorithmically rather than restoring separators from exact sites.

`RemoveTokenPlan` records both semantic visible siblings and the exact removed `TOKEN` site:

```ts
type RemoveTokenPlan = {
  token: HTMLElement;
  insertionSite: InsertionSite;
  separatorBefore: Text | null;
  separatorAfter: Text | null;
  previousVisibleSibling: HTMLElement | null;
  nextVisibleSibling: HTMLElement | null;
};
```

`DeleteHighestEmptySubtreePlan` records the removed subtree and where it came from:

```ts
type DeleteHighestEmptySubtreePlan = {
  highest: Element | null;
  insertionSite: InsertionSite | null;
};
```

The first inverse functions are intentionally local and strict-LIFO:

```ts
undoDeleteHighestEmptySubtree(plan.deleteHighestEmptySubtreePlan);
undoRemove(plan.removePlan);
```

`executeDelete` may insert a maintenance `ANCHOR`, but the plan does not currently remember it:

```ts
if (!plan.prevCrs && !plan.nextCrs) {
  const anchor = token.createAnchor();
  // insert anchor, then place cursor
}
```

Either record `insertedAnchor` during execution, or define a safe rule for finding and removing the maintenance anchor during undo.
