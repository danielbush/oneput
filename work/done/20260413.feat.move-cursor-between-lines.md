# feat: move CURSOR between LINE's

## Problem

The CURSOR currently works within a single `LINE`, but the first item on the jsed critical path is broader inter-`LINE` cursor movement. Once the user has exhausted the current `LINE`, jsed should be able to find the next editable `LINE` and place the CURSOR on its first editable `LINE_SIBLING`.

This work also exposes a mismatch in `quickDescend`: the current contract is framed around "first `TOKEN`", but inter-`LINE` movement and Enter semantics want "first real CURSOR target". In practice that means descending through structural containers until we find a `TOKEN` or an `ISLAND`, rather than stopping on arbitrary non-terminal `LINE_SIBLING` containers.

## Design

Refine inter-`LINE` movement in two layers:

1. `quickDescend(...)` should return the first CURSOR target in the reachable subtree: a `TOKEN` or `ISLAND`, not just the first `TOKEN`.
2. When the CURSOR exhausts the current `LINE`, it should:
   - use `findNextNode(...)`
   - find the next relevant text node or `TOKEN`
   - resolve that node to its owning `LINE`
   - call `quickDescend(...)` on that `LINE`
   - place the CURSOR on the returned `TOKEN` or `ISLAND`

This should make cross-`LINE` movement explicit without requiring eager tokenization of the whole document.
