# feat: ancestor-chain-aware navigation

## Problem

The current navigation model does not preserve enough context when moving up and down the FOCUSABLE tree. That makes vertical movement feel lossy: once the user moves upward, returning downward may not reliably bring them back to the structurally corresponding place they came from.

There is a design idea to distinguish:

- vertical movement through ancestor/descendant structure
- horizontal movement across siblings

The intended behavior is that UP/DOWN remembers a chain through the tree, while PREV/NEXT SIBLING changes that chain.

## Questions

- What should the remembered chain be anchored to?
- When exactly should sibling navigation invalidate or replace the remembered chain?
- How should the persisted marker behave if the document structure changes between sessions?
- Should the restored location be treated as a preferred target or a hard requirement?

## Solution

At the moment FOCUS tracks the current FOCUSABLE element and CURSOR tracks the current TOKEN when we're in edit mode.

We introduce a third thing to track: a remembered FOCUSABLE called the CURRENT_MARK.

The CURRENT_MARK is the lowest remembered point in the current structural chain.

Rules for updating CURRENT_MARK:

- moving UP does not update CURRENT_MARK
- moving further down updates CURRENT_MARK to the new FOCUS
- moving to another sibling updates CURRENT_MARK to the new FOCUS

`$mod+l` has two modes.

### Chain-follow mode

If FOCUS is an ancestor of CURRENT_MARK, `$mod+l` should move down toward CURRENT_MARK.

This does not need a tree walk.

Instead:

- walk up `parentNode` from CURRENT_MARK
- check whether FOCUS appears on that ancestor chain
- if it does, move FOCUS to the next node down from it on that same chain

This means repeated `$mod+l` walks back down the remembered chain until we reach CURRENT_MARK.

### Subtree mode

If FOCUS is not an ancestor of CURRENT_MARK, `$mod+l` should behave as a pre-order walk within the current FOCUS subtree.

That includes the case where FOCUS is already at CURRENT_MARK.

In this mode:

- use `walk.findNextNode(...)`
- allow normal pre-order descent inside the current FOCUS subtree
- do not walk outside that subtree
- if a next FOCUSABLE is found, move there and update CURRENT_MARK
- if no next FOCUSABLE is found, do nothing

Other bindings:

- `$mod+h` moves UP via `Nav.UP`
- `$mod+j` moves to the next FOCUSABLE sibling
- `$mod+k` moves to the previous FOCUSABLE sibling

This gives `$mod+h` / `$mod+l` a stable vertical-chain meaning, while `$mod+j` / `$mod+k` switch to a different sibling branch and redefine the CURRENT_MARK.

## Initial Proposal

- `cmd+h,l` should be UP/DOWN
- `cmd+j,k` should be PREV/NEXT SIBLING
- we should remember the ancestor chain
- if you go up and down, we keep the same chain
- changing sibling updates the chain
- the chain can be represented by the lowest element; so going DOWN extends the existing chain; UP does not update anything
- leave a permanent marker in the DOM pointing to the lowest element in the chain; when the editor loads again, it can jump automatically to this point
- `$mod+shift+h,l` does pre-order walk
