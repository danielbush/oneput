# refactor: DOM type unification

## Problem

Functions in `packages/jsed/src/lib/` use an inconsistent mix of DOM types in their signatures: `ChildNode`, `ParentNode`, `HTMLElement`, `Element`, `Node`, `EventTarget`, and unions of these. This causes friction — frequent casts, union types that don't narrow well, and no clear convention for what to use where.

## Goal

Settle on a consistent convention and apply it. The likely convention is:

- **`HTMLElement`** as the default for jsed's own internal functions (parameters and return types). This is what jsed actually works with — it reads classList, tagName, style, etc.
- **Narrower DOM types** (`Node`, `EventTarget`, etc.) only at genuine boundaries — e.g. DOM API callbacks that hand you a `Node`, or event handlers that receive `EventTarget`.
- **`null`** unions where the function legitimately might receive or return null (e.g. traversal functions that walk off the edge).

## How to do this

1. Read `packages/jsed/CLAUDE.md` and the `/jsed` skill to understand the codebase structure.

2. **Audit phase** — scan all exported functions in `packages/jsed/src/lib/` (the key files are `traversal.ts`, `token.ts`, `dom-rules.ts`). For each function, note:
   - Current parameter types and return type
   - What DOM properties/methods the function actually uses on those parameters
   - Whether the current type is too broad (e.g. `Node` when it reads `classList`) or too narrow

3. **Propose a convention** — write a brief summary of what you plan to change and why. Don't change anything yet. Output this as a comment at the top of your work so the human can review your reasoning.

4. **Apply changes** — update function signatures to match the convention. Work file by file. After each file:
   - Fix any resulting type errors in that file and its callers
   - Run `task check` to verify type checking passes
   - Run `task test` to verify all tests pass

5. **Do NOT change behaviour** — this is a pure type refactor. No logic changes, no new functions, no deleted functions. If a cast is genuinely needed at a boundary (e.g. DOM API returns `Node` but we know it's `HTMLElement`), keep it — just make sure it's at the boundary, not scattered through internal code.
