# feat: cursor transparent by default

## Problem

`LOOSE_LINE` handling currently exposes an awkward structural model to the `CURSOR`: some real `LINE`-ish content reads like an `OPAQUE_BLOCK` stop that the `CURSOR` moves onto but not into. That is jarring during ordinary cursor movement and gets stranger when `TokenSelection` grows across the same structure.

The likely root issue is that jsed still treats too many non-`ISLAND` structures as places the `CURSOR` should land on rather than descend through. That makes the navigation model harder to explain and forces more special-case reasoning around line crossing, token access, and selection growth.

## Proposed direction

Bias the taxonomy toward `CURSOR_TRANSPARENT` by default:

- `ISLAND` stays `CURSOR_OPAQUE` and `FOCUS`-opaque as today.
- Other non-`ISLAND` structures should generally be descended through by the `CURSOR`, not visited as stops.
- If we still need true `CURSOR_OPAQUE` elements outside `ISLAND`, make that an explicit opt-in concept rather than the ambient default.

This should make `LOOSE_LINE` content feel like part of the surrounding editable flow instead of a separate stop, and should simplify how selections reason about contiguous runs of editable content.

## Open questions

- Do we need an explicit `CURSOR_OPAQUE` concept distinct from `ISLAND`, or can `ISLAND` remain the only opaque case for now?
- Should `TokenCursor` become token-only again, or does it still need to sit on non-`TOKEN` `LINE_SIBLING` targets such as `ISLAND` and any future explicit `CURSOR_OPAQUE` nodes?
- What vocabulary best distinguishes `CURSOR_OPAQUE` from `ISLAND` without duplicating meaning?
