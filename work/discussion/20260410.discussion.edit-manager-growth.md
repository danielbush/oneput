# discussion: handling `EditManager` growth

## Problem

`EditManager` is the orchestration layer for jsed. That is the right place for:

- edit-session lifecycle
- coordinating `Nav`, `TokenCursor`, and `UserInput`
- translating user gestures into editor actions
- publishing state changes back out to the app layer

But as more spacing, anchor, and menu-related behaviour is added, the module risks becoming a grab-bag of:

- key-oriented handlers
- semantic editor actions
- low-level DOM mutation details
- notification plumbing

That makes it harder to keep `EditManager` readable as a deep module.

## Current understanding

The healthiest direction seems to be:

- keep `EditManager` as an application-service / orchestration module
- keep key-oriented handlers thin
- push low-level DOM mechanics into `lib/`, `TokenCursor`, and `Nav`
- introduce semantic helper methods under the public gesture handlers
- use explicit hooks to publish important state transitions without giving `EditManager` menu/UI knowledge

This has already started in a few places:

- `handleInputChange(...)` now uses a logic-sandwich shape with `decideInputIntent(...)`
- `handleEnter()` now delegates to semantic helpers like `splitAtCursor()`
- explicit hooks now exist for:
  - `onModeChange`
  - `onFocusChange`
  - `onCursorChange`
  - `onTextChange`
  - `onElementChange`

## Candidate patterns

### 1. Logic sandwich

Use for methods that currently mix:

- state gathering
- decision logic
- DOM / cursor effects

Shape:

- top: gather state from the world
- middle: compute intent / action
- bottom: apply effects

This keeps orchestrator methods readable and lets the decision layer become directly testable.

### 2. Semantic helper extraction

Keep public gesture handlers named after user input:

- `handleEnter()`
- `handleLeft()`
- `handleRight()`

But move real editor semantics into helpers such as:

- `enterEditingAtFocus()`
- `enterEditingAtTarget(...)`
- `splitAtCursor()`
- `insertAnchorAfterFocusedTag()`

That keeps the top-level module understandable to both humans and agents without forcing them to read large key-handler bodies.

### 3. Policy vs mechanism

`EditManager` should own policy:

- what should happen now?
- which collaborator should do it?

It should not own low-level DOM mechanisms where avoidable.

Those belong in:

- `lib/token.ts`
- `TokenCursor`
- `Nav`

### 4. Explicit hook publishing

Instead of a single vague “something changed” callback, use explicit hooks for meaningful editor-state transitions:

- `onModeChange`
- `onFocusChange`
- `onCursorChange`
- `onTextChange`
- `onElementChange`

This lets app code decide whether menu/UI state should be recomputed, while keeping `EditManager` free of menu concepts.

### 5. Split by workflow, not by size

If `EditManager` continues to grow, split it by coherent workflow rather than arbitrary line count.

Concrete candidate sub-services:

- `EditSessionLifecycle`
  - enter/exit editing
  - connect/disconnect input subscriptions
  - initialize the first CURSOR target
- `EditInputInterpreter`
  - interpret `UserInputChange`
  - produce semantic edit intents
- `EditStructureActions`
  - split/join/anchor/space actions
  - other structural editor operations that coordinate `TokenCursor` and `lib/token`
- `EditNotifications`
  - emitting `onModeChange`
  - `onFocusChange`
  - `onCursorChange`
  - `onTextChange`
  - `onElementChange`

The split should preserve `EditManager` as the shallow top-level surface and move details under it.

### 6. Command-style actions

For nontrivial operations, introduce small semantic action methods or action objects rather than letting complex workflows live inline inside broad handlers.

Examples:

- `insertSpaceAfterTag`
- `removeSpaceBeforeTag`
- `insertAnchorAfterTag`
- `splitAtCursor`

These actions can stay as methods or become deeper helper objects later, but the key point is the same:

- name the semantic editor operation explicitly
- keep the top-level orchestration readable
- make the operation easier to test in isolation

## Concrete direction

Short term:

- continue extracting semantic helpers under key-oriented handlers
- continue pushing DOM/boundary rules into `lib/`
- keep `EditManager` as the place where collaborators are coordinated

Medium term:

- identify one or two dense workflow clusters and move them into deeper helper modules or sub-services
- keep top-level `EditManager` readable as the orchestration entry point

## Questions

- Which remaining `EditManager` methods still mix gesture handling and semantic action too tightly?
- Is the next extraction seam input interpretation, structural edit actions, or session lifecycle?
- Should `EditManager` stay as one top-level module with deeper helper modules, or should some workflows become standalone application-service classes?

## Initial Proposal

- Treat `EditManager` as an application-service/orchestrator
- Keep public gesture handlers thin
- Extract semantic operations beneath them
- Use logic sandwich where state gathering, decision logic, and effects are currently mixed
- Push low-level DOM mechanics downward and keep top-level code easy to scan
