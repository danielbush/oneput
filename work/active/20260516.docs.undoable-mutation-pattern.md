# docs: Undoable mutation pattern

The first undoable mutation work targeted `Cursor.delete()` and treated it as the public semantic operation boundary. Call sites such as `EditorOps` still call `cursor.delete(...)` normally; the public cursor method is decorated with `@recordUndo`, and the decorator records whatever semantic result the method returns.

The undo dependency is injected through `CursorParams` into `CursorState` as an optional `undo?: UndoRecorder`. Production editor cursors receive the editor state's recorder. Internal cursors, such as the silent cursor used by `CursorSelection`, can omit undo because they are not the semantic mutation owner. `Cursor` forwards decorated operation results to the optional recorder through `_recordUndoResult`.

`UndoRecorder.record()` owns the decision about whether a result should enter history. Operation results carry an explicit `undoable` flag. Successful delete results use `undoable: true`; no-op results use `undoable: false`. This keeps `@recordUndo` operation-agnostic and avoids naming conventions such as checking for `-noop` suffixes.

Lower-level mutating helpers were changed to return semantic facts rather than undo entries. `token.remove()` now returns a `TokenRemovalResult` with the removed `TOKEN`, its insertion site, previous/next visible siblings, and any removed separator text nodes with their sites. `token.insertBefore`, `token.insertAfter`, and `token.append` return `TokenInsertionResult`. `token.removeParent` returns `TokenParentRemovalResult`. `deleteEmptyTree()` returns `DeleteEmptyTreeResult` with every removed empty ancestor and its original insertion site.

`CursorTextOps.delete()` stitches those lower-level results into one atomic `CursorDeleteResult`. It records the requested deletion type, the removed token result, any inserted `ANCHOR`, any removed parent, any pruned empty tree, the `CURSOR` before and after the operation, and the input cursor positioning decision. Tokenization and `state.place(...)` are treated as editor-maintenance effects rather than undo-relevant document mutations.

The current implementation stops at producing and recording the semantic object. The actual undo/redo implementation still needs to consume this result, restore removed ancestors and parent nodes, restore removed separators and the deleted `TOKEN`, remove any inserted `ANCHOR`, and decide where to place the `CURSOR` after undo or redo.
