# refactor: Default to OPAQUE_BLOCK for nested FOCUSABLE's

Currently any NON_INLINE_FLOW, non-ISLAND FOCUSABLE defaults to TRANSPARENT_BLOCK — the CURSOR descends into it seamlessly. OPAQUE_BLOCK requires an explicit `jsed-cursor-opaque` class. This discussion proposes flipping the default.

## Motivation

### Clearer editing model

With opaque-by-default, the LINE is a firm editing boundary. The CURSOR stays scoped to one LINE; Nav handles moving between LINE's. This is predictable and matches the phone experience — you tap a different paragraph to edit it, you don't arrow into it.

### SHALLOW_TOKENIZATION edge case

With transparent-by-default, `tokenizeLine` recurses through all TRANSPARENT_BLOCK descendants. Even with quick-descend as the entry point (so the initial `tokenizeLine` targets the right LINE), the CURSOR's `onEnterBlockTransparent` callback can trigger a cascade:

```html
<p>
  some text
  <div>                      <!-- TRANSPARENT_BLOCK → lazyTokenize -->
    <div>                    <!-- TRANSPARENT_BLOCK → tokenizeLine recurses -->
      <div>                  <!-- recurses again -->
        ...lots of content...
      </div>
    </div>
  </div>
</p>
```

One arrow press past the last TOKEN hits the nested `<div>`, `onEnterBlockTransparent` fires `lazyTokenize`, and `tokenizeLine` recurses through the entire transparent subtree. With opaque-by-default, the CURSOR lands on that `<div>` as an opaque LINE_SIBLING — no cascade.

## Quick-descend assumption

The original objection to opaque-by-default was extra friction — the user FOCUS's on a container, then must "enter" it to edit TOKEN's. This is resolved by a **quick-descend** function: when the user FOCUS's on a FOCUSABLE (via Nav or touch), quick-descend finds the first text node or TOKEN, establishes the LINE, and enters edit mode directly. This makes opaque-by-default no worse than transparent-by-default for the common "go fix a word" case.

Touch (REQUEST_FOCUS) already bypasses the opaque boundary by targeting the LINE directly.

## Implementation plan

### 1. Constants (`src/lib/constants.ts`)

- Add `JSED_CURSOR_TRANSPARENT_CLASS = 'jsed-cursor-transparent'`
- Remove or deprecate `JSED_CURSOR_OPAQUE_CLASS`

### 2. Predicates (`src/lib/traversal.ts`)

- **`isCursorBoundary`** — flip: a LINE is a cursor boundary unless it has `jsed-cursor-transparent` class
- **`isTransparentBlock`** — becomes `isLine(el) && el.classList.contains(JSED_CURSOR_TRANSPARENT_CLASS)`
- **`isLineSibling`** — include any LINE that is not transparent (i.e. `isLine(el) && !isTransparentBlock(el)`)

### 3. Tokenization (`src/lib/token.ts`)

No logic change — `tokenizeLineRec` already gates on `isTransparentBlock(child)`. The flipped predicate narrows tokenization automatically.

### 4. Tests

- lineSibling and TokenCursor tests that use `jsed-cursor-opaque` flip to `jsed-cursor-transparent` on elements that should be transparent
- Tests relying on the default being transparent need adjusting — nested blocks in fixtures will now be opaque by default

### 5. Docs

- `vocabulary.md` — swap which is the default in OPAQUE_BLOCK / TRANSPARENT_BLOCK descriptions
- `architecture.md` — minor wording update

### Not affected

- **Nav** — doesn't know about opaque/transparent, only sees FOCUSABLE's
- **InputManager / EditManager** — delegate to TokenCursor which delegates to traversal predicates
