# fix: re-establish reliable scrollIntoView behavior

## Problem

Follow-up to `SCROLL_INTO_VIEW__WORK`.

- add `scrollIntoViewIfHidden` alongside `scrollIntoViewIfSmaller`
  - it should scroll a FOCUSABLE or TOKEN into view if partially or completely outside the viewport
  - it should use a "nearest" strategy to bring the element back into view with the least amount of scrolling
- remove `params?: { scrollIntoView?: boolean }` from FOCUS, REQUEST_FOCUS, `#updateFocus` etc in Nav
  - we want scrolling to be automatic
- `#updateFocus` should call `scrollIntoViewIfHidden` instead
- setToken in TokenCursorBase should also call `scrollIntoViewIfHidden`
- add an action to EditDocument using cmd+m that runs either scrolls the CURSOR's current TOKEN or if that is not avaiable, the current FOCUS, to the center of the screen
  - if scrolling the FOCUS an it's larger than the viewport, scroll the top edge into view at the top of the viewport
- I think we can remove `scrollIntoViewIfSmaller` also

## Changes

- **2026-04-09** — created as an active follow-up to `SCROLL_INTO_VIEW__WORK` because the earlier closed ticket no longer fully captures the current problem
- **2026-04-09** — replaced `scrollIntoViewIfSmaller` with automatic hidden-only scrolling for FOCUS and CURSOR, removed Nav's opt-out scroll flags, and added a `cmd+m` reveal action that centers the current TOKEN or reveals the current FOCUSABLE from its top edge when oversized
