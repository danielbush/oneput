# feat: quick-descend into a FOCUSABLE

## Problem

`TokenManager.tokenize` returns the first LINE_SIBLING, which may be an OPAQUE_BLOCK (e.g. a `<p>` inside a `<div>`). The CURSOR lands on the `<p>` itself rather than descending into it to find TOKEN's. FOCUS also stays on the outer `<div>` rather than moving to the LINE being edited.

## Design

Quick-descend walks from a FOCUSABLE inward to find the first LINE that has editable content — TOKEN's, INLINE's (like `<em>`), or INLINE_ISLAND's (like inline katex). If the first LINE_SIBLING is an OPAQUE_BLOCK, recurse into it. Keep walking until a qualifying LINE is found.

A qualifying LINE is one that, when tokenized, would produce TOKEN's. If no qualifying LINE exists (e.g. the FOCUSABLE only contains other OPAQUE_BLOCK's with no text), return null — the user stays in view mode and can FOCUS on individual elements via Nav.

Once found:
- FOCUS moves to the qualifying LINE (so ElementIndicator shows the right tag)
- CURSOR_LINE is set to that LINE
- CURSOR lands on the first TOKEN

## Implementation

Modify `TokenManager.tokenize`: when `getFirstLineSibling` returns a non-TOKEN (OPAQUE_BLOCK), recurse into it rather than returning it directly. Also update `EditManager.getFirstTokenUnderFocus` to derive LINE and FOCUS from the returned token rather than from the original focus element.
