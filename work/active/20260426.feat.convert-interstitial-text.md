# feat: convert all interstitial text before jsed operates on the document

## Problem

`LOOSE_LINE` is better described as **interstitial text** — text and inline runs that sit around `NESTED_LINE`'s within an outer `LINE`. The current handling forces a lot of complicated logic in shallow tokenization and cross-line traversal (especially in the previous direction), because the interstitial runs are discovered and tokenized opportunistically via `tokenizeLooseLinesIn` / `tokenizeLooseLinesAround`.

If we instead wrap every interstitial run in an inline span at the start of the session — treating each as a kind of implicit line rather than an `INLINE_FLOW` — tokenization can be uniform node-based work, and cross-line walks can stop reasoning about loose runs as a special case.

## Proposed direction

- At session start (or document load), walk the document and wrap each interstitial run in a span. The span is **not** an `INLINE_FLOW`; it is a new kind of implicit line so the `CURSOR` treats it as part of the outer flow without descending it as inline.
- "Interstitial" does not apply to text around inline-ish elements (including `inline-block`'s) — those belong to the surrounding `INLINE_FLOW`/`LINE` already.
- Anchor actions need to be updated so that adding an `ANCHOR` into an interstice happens inside an interstitial span.
- Once interstitials are explicit, switch tokenization to plain node-based work and drop `tokenizeLooseLinesIn` / `tokenizeLooseLinesAround`.
- Bonus UX: because interstitials are now first-class, we can offer to convert the interstitial under `FOCUS` (or all of them) into a proper `LINE` such as a `<p>`. This helps the user navigate and rectify bad HTML.

## Open questions

- What's the right vocabulary term for the wrapper? It is implicit-line-ish but distinct from `IMPLICIT_LINE` (which promotes to a real wrapping `LINE`) and from `INLINE_FLOW`.
- Does this actually simplify `findPreviousCrossLineTarget`, or only tokenization? Initial read: it simplifies tokenization; cross-line may benefit indirectly via uniform node-based seats.
- Interaction with the `CURSOR_TRANSPARENT_BY_DEFAULT__WORK` taxonomy change — sequencing matters; interstitials should land first per the comment in the backlog.

- [x] 2. `<br>` between text — `<div>aaa<br>bbb<p>...</p></div>`. `<br>` isn't in INLINE_TAGS, so it splits the run. Likely correct (a `<br>` is a line break) but worth pinning.
- [x] 3. Comment nodes — `<!-- -->` between text. Currently breaks runs. Probably fine, but pinned via test.
- [x] 4. Inline `<img> / <input>` — currently treated as block boundaries. Likely wrong — they should be absorbed into the interstitial. Worth a test that drives the fix.
- [x] 5. Run with only inline elements, no text — e.g. `<div><em>x</em><p>y</p></div>`. Should the em be wrapped? Probably yes (so the cursor reaches it), but pin it.
