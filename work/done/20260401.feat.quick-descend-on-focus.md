# feat: quickDescend on FOCUS

## Problem

The current "focus twice to edit" flow delays `quickDescend` until the second click/touch on the same FOCUSABLE. That preserves a clean separation between FOCUS and CURSOR, but it means the first click does not prepare the target for TOKEN-level hit testing.

We want first FOCUS to eagerly tokenize the FOCUSABLE so a second click/touch can select a TOKEN directly. The interaction still needs to remain two-step:

- First click/touch on a new FOCUSABLE: update FOCUS and run `quickDescend`, but do not open the CURSOR
- Second click/touch on that same FOCUSABLE: enter edit mode and allow TOKEN selection

The same rule must hold for FOCUSABLE's that were already tokenized earlier: clicking a different already-tokenized FOCUSABLE should only update FOCUS; the next click/touch on that same FOCUSABLE should be the one that selects a TOKEN.

## Design

Keep the current separation between navigation and editing, but move SHALLOW_TOKENIZATION earlier in the flow.

### Interaction model

1. `REQUEST_FOCUS` on a different FOCUSABLE updates FOCUS and opportunistically `quickDescend`s that FOCUSABLE
2. If `quickDescend` finds a TOKEN, jsed leaves the result in the DOM but does not create or move the CURSOR
3. A second click/touch on the already-focused FOCUSABLE enters edit mode and resolves the clicked TOKEN from the already-tokenized DOM rather than defaulting to the first TOKEN

This preserves the existing "focus twice to edit" mental model while making the second click more precise.

### Constraints

- Do not auto-open the CURSOR during first FOCUS
- Do not change FOCUS to the descended LINE during first FOCUS; first click is still structural navigation
- If `quickDescend` returns null, first click should still just set FOCUS
- If a FOCUSABLE is already tokenized, focusing it from elsewhere should remain navigation-only; only the second click/touch on the same focused element should enter TOKEN editing
- First-focus tokenization must be safe to repeat or skip; revisiting an already-tokenized FOCUSABLE must not duplicate TOKEN wrappers or otherwise perturb the DOM
- Touch should follow the same two-step rule as click

## Acceptance criteria

- [ ] First click/touch on an untokenized FOCUSABLE sets FOCUS and leaves tokenized DOM behind, but does not open the CURSOR
- [ ] Second click/touch on that same focused FOCUSABLE opens the CURSOR on the TOKEN hit by the interaction
- [ ] If the second click/touch on the focused FOCUSABLE does not hit a TOKEN, edit mode falls back to the first TOKEN from `quickDescend`
- [ ] First click/touch on a different FOCUSABLE that was tokenized earlier only changes FOCUS
- [ ] Second click/touch on that newly focused FOCUSABLE opens the CURSOR on the TOKEN hit by the interaction
- [ ] Re-focusing an already-tokenized FOCUSABLE does not re-wrap or duplicate TOKEN markup
- [ ] `quickDescend` returning null leaves jsed in view mode with only FOCUS updated
