# refactor: Tripartite delete pattern

This ticket records the pattern used in `git diff main..undo-attempt-2` to transform an existing imperative mutation like `CursorTextOps.delete()` into the tripartite form:

```ts
const plan = planX(...);
executeX(..., plan);
return plan;
```

Use `delete` as the worked example. The goal is not to finish undo/redo here; it is to make an existing operation return a semantic plan that the undo recorder can observe later.

## Approach

Start with the existing imperative operation and keep its behavior intact. In `CursorTextOps.delete()`, the original shape was roughly:

```ts
delete({ type }: CursorDeleteOpts = { type: 'tokenDeletion' }) {
  if (!this.state.isOnToken()) return;

  const current = this.state.getPlace();
  const prevCrs = this.state.motion.getPrevious();
  const nextCrs = this.state.motion.getNext();
  const parentNode = current.parentNode as HTMLElement;
  const [prevSibling, nextSibling] = token.remove(current);

  let inputCursorPosition: UserInputOpts['inputCursorPosition'] = 'end';
  if (type === 'tokenDeletion' || !prevCrs) {
    inputCursorPosition = 'selectAll';
  }

  if (!prevCrs && !nextCrs) {
    const anchor = token.createAnchor();
    // insert anchor near the deletion site
    this.state.place(anchor, { inputCursorPosition });
    return;
  }

  if (!prevSibling && !nextSibling) {
    token.removeParent(parentNode);
    deleteHighestEmptyTree(parentNode.parentNode as HTMLElement, this.state.document.root);
  }

  this.state.place((prevCrs || nextCrs) as HTMLElement, { inputCursorPosition });
}
```

Then split the operation into three parts.

1. The public wrapper remains small and returns the semantic plan:

```ts
delete(opts?: CursorDeleteOpts): CursorDeletePlan {
  const plan = planDelete(this.state, opts);
  executeDelete(this.state, plan);
  return plan;
}
```

2. `planDelete` gathers the facts needed before mutation. It does not mutate the DOM. It composes lower-level plans, such as `planRemove(current)` and `planDeleteHighestEmptyTree(...)`.

```ts
export function planDelete(
  state: CursorState,
  { type }: CursorDeleteOpts = { type: 'tokenDeletion' }
): CursorDeletePlan {
  if (!state.isOnToken()) {
    return { type: 'delete-noop', undoable: false };
  }

  const current = state.getPlace();
  const prevCrs = state.motion.getPrevious();
  const nextCrs = state.motion.getNext();
  const parentNode = current.parentNode as HTMLElement;
  const inputCursorPosition = type === 'tokenDeletion' || !prevCrs ? 'selectAll' : 'end';
  const removePlan = planRemove(current);
  const removeParent = !removePlan.previousVisibleSibling && !removePlan.nextVisibleSibling;

  return {
    type: 'delete',
    undoable: true,
    deletionType: type,
    current,
    prevCrs,
    nextCrs,
    parentNode,
    removePlan,
    removeParent,
    deleteHighestEmptyTreePlan: removeParent
      ? planDeleteHighestEmptyTree(parentNode.parentElement, state.document.root, {
          withoutChild: parentNode
        })
      : null,
    inputCursorPosition
  };
}
```

3. `executeDelete` performs the side effects from the plan. It returns `void`; the plan is already owned by the caller.

```ts
export function executeDelete(state: CursorState, plan: CursorDeletePlan): void {
  if (plan.type === 'delete-noop') {
    return;
  }

  executeRemove(plan.removePlan);

  const prevSibling = plan.removePlan.previousVisibleSibling;
  const nextSibling = plan.removePlan.nextVisibleSibling;
  const userInputOpts: UserInputOpts = { inputCursorPosition: plan.inputCursorPosition };

  if (!plan.prevCrs && !plan.nextCrs) {
    const anchor = token.createAnchor();
    if (prevSibling) {
      token.insertAfter(anchor, prevSibling);
    } else if (nextSibling) {
      token.insertBefore(anchor, nextSibling);
    } else {
      token.append(anchor, plan.parentNode);
    }
    state.place(anchor, userInputOpts);
    return;
  }

  if (plan.removeParent) {
    token.removeParent(plan.parentNode);
    if (plan.deleteHighestEmptyTreePlan) {
      executeDeleteHighestEmptyTree(plan.deleteHighestEmptyTreePlan);
    }
  }

  state.place((plan.prevCrs || plan.nextCrs) as HTMLElement, userInputOpts);
}
```

The top-level plan type lives beside the operation API, because it is part of what callers can observe:

```ts
export type CursorDeletePlan =
  | {
      type: 'delete-noop';
      undoable: false;
    }
  | {
      type: 'delete';
      undoable: true;
      deletionType: CursorDeleteOpts['type'];
      current: HTMLElement;
      prevCrs: HTMLElement | null;
      nextCrs: HTMLElement | null;
      parentNode: HTMLElement;
      removePlan: RemoveTokenPlan;
      removeParent: boolean;
      deleteHighestEmptyTreePlan: DeleteHighestEmptyTreePlan | null;
      inputCursorPosition: UserInputOpts['inputCursorPosition'];
    };
```

Lower-level mutations should be transformed first when the top-level operation depends on their pre-mutation facts. For `delete`, `token.remove()` became:

```ts
export type RemoveTokenPlan = {
  type: 'remove-token';
  token: HTMLElement;
  separatorBefore: Text | null;
  separatorAfter: Text | null;
  previousVisibleSibling: HTMLElement | null;
  nextVisibleSibling: HTMLElement | null;
};

export function planRemove(token: HTMLElement): RemoveTokenPlan {
  return {
    type: 'remove-token',
    token,
    separatorBefore: getSeparatorBefore(token),
    separatorAfter: getSeparatorAfter(token),
    previousVisibleSibling: getPreviousVisibleSibling(token),
    nextVisibleSibling: getNextVisibleSibling(token)
  };
}

export function executeRemove(plan: RemoveTokenPlan): void {
  plan.token.remove();
  // Apply separator cleanup rules from the recorded candidate separators.
}

export function remove(token: HTMLElement): RemoveTokenPlan {
  const plan = mut.planRemove(token);
  mut.executeRemove(plan);
  return plan;
}
```

Not every DOM change inside execution needs to become its own subplan immediately. For the current exploratory stage, keep derived maintenance behavior rule-based when it can be recomputed from the plan.

Separators are the first example. `RemoveTokenPlan` records the candidate whitespace nodes around the `TOKEN`, but it does not record a step-by-step script of which separator to delete. `executeRemove` applies the separator rules algorithmically:

```ts
export function executeRemove(plan: RemoveTokenPlan): void {
  plan.token.remove();

  if (plan.separatorBefore?.parentNode && plan.separatorAfter?.parentNode) {
    plan.separatorAfter.remove();
  }

  if (plan.separatorBefore?.parentNode) {
    if (!getNextVisibleNodeSibling(plan.separatorBefore)) {
      plan.separatorBefore.remove();
    }
    if (plan.separatorBefore.parentNode && !getPreviousVisibleNodeSibling(plan.separatorBefore)) {
      plan.separatorBefore.remove();
    }
  }
}
```

Anchors are the second example. When deleting the last reachable `TOKEN`, `executeDelete` can derive the required `ANCHOR` insertion from the delete plan instead of storing a separate anchor plan:

```ts
if (!plan.prevCrs && !plan.nextCrs) {
  const anchor = token.createAnchor();
  if (prevSibling) {
    token.insertAfter(anchor, prevSibling);
  } else if (nextSibling) {
    token.insertBefore(anchor, nextSibling);
  } else {
    token.append(anchor, plan.parentNode);
  }
  state.place(anchor, userInputOpts);
  return;
}
```

This keeps the plan focused on semantic destructive changes: the removed `TOKEN`, surrounding context, and empty tree pruning. Separator cleanup and anchor insertion are currently execution rules derived from that context. Revisit this boundary when undo/redo needs to remove inserted anchors or restore whitespace precisely; at that point the plan may need execution-produced fields such as `insertedAnchor`.

`deleteHighestEmptyTree` follows the same wrapper rule:

```ts
export function deleteHighestEmptyTree(
  el: Element,
  ceiling?: Element
): mut.DeleteHighestEmptyTreePlan {
  const plan = mut.planDeleteHighestEmptyTree(el, ceiling);
  mut.executeDeleteHighestEmptyTree(plan);
  return plan;
}
```

## Rules to repeat

- Keep the public operation name as the compatibility wrapper.
- Add `planX` to collect all facts that must be known before mutation.
- Add `executeX` to apply side effects from the plan and return `void`.
- Return the plan from the public wrapper so decorators and undo infrastructure can record it.
- Use named plan fields, not tuples or iterators.
- Compose lower-level plans inside higher-level plans instead of flattening everything too early.
- Keep separators and anchors rule-based for now when their behavior can be derived from the recorded plan context.
- Preserve the old imperative body as a nearby comment during exploration
