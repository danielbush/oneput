# discussion: FOCUS on the document root

## Situation

Walk functions use the jsed document root as the ceiling, which means by default the root is never visited during traversal. At startup, Nav sets the FOCUS to the document root. From there the user navigates down into the first FOCUSABLE.

The `visitCeiling` parameter exists in the walk module and would allow the root to be visited, but it's off by default. So once the user moves FOCUS off the root, they can never return to it via navigation.

## Edge case: empty document

An empty document has no child FOCUSABLE's. The simplest thing is to let FOCUS sit on the root and allow the user to start adding elements via the Oneput menu. This works today — Nav starts with FOCUS on the root and there's nowhere to go.

The temptation was to have Nav auto-descend at startup to find the first child FOCUSABLE. But that breaks the empty document case — there's nothing to descend into, and the root needs to remain the FOCUS target.

## Current state

- FOCUS starts on the document root
- Once the user navigates away, they can't get back
- `visitCeiling` exists but is never turned on for this purpose
- This is... probably fine?

## Thinking (26-Mar-2026)

Initial lean was towards leaving it as-is — the root as a startup landing spot, document-level actions in the Oneput menu.

But a better approach: **eliminate the empty document case entirely**. If we always ensure the document has at least one FOCUSABLE, then auto-descend at startup becomes unconditional and the "FOCUS on root" special case disappears. The root becomes purely a ceiling — never focused, never visited, no need for `visitCeiling`.

### Startup logic

1. If the document is empty, inject a seed element
2. Auto-descend to the first FOCUSABLE
3. No special case for root

### What seed element?

- **Empty `<p>`** — minimal, gets an ANCHOR via `addAnchors`, user starts typing immediately. Feels natural for a blank document.
- **Template with structure** — more opinionated, could be useful if documents tend to follow a pattern. Probably too prescriptive for a general-purpose editor.
- **Configurable** — let the consumer (e.g. `jsed-demo`) provide a seed factory. jsed itself just guarantees "never empty".

Leaning towards an empty `<p>` as default with an optional seed override. jsed's contract becomes: the document always has at least one FOCUSABLE, and the root is never the FOCUS target.

### What this simplifies

- No special "FOCUS on root" state at startup
- No question about whether `visitCeiling` should be on
- Nav traversal is uniform — always operates on FOCUSABLE's within the root, never on the root itself
- Document-level actions (adding elements, changing properties) live in the Oneput menu
