# refactor: detokenize on the fly

## Problem

Jsed currently tokenizes `LINE`s on demand, but it does not yet have a complementary detokenization pass. Over a longer editing session, that means more and more of the document stays tokenized, which increases DOM node count and keeps jsed farther away from the original HTML shape than necessary.

The critical-path need is to reclaim old tokenization work without breaking active editing. In practice that means we need a way to remember which `LINE`s were tokenized, observe when tokenization happens, and detokenize only `LINE`s that are no longer occupied by a `CURSOR`.

## Design

Introduce a `Tokenizer` service that owns a `Detokenizer`, and have `EditManager` / `TokenCursor` go through that service.

The current intended flow is:

1. `Tokenizer` becomes the canonical entry point for tokenization operations such as `quickDescend(...)`.
2. `Tokenizer` owns `Detokenizer`.
3. `Detokenizer` records which `LINE`s have become tokenized and in what order.
4. It knows enough about active `CURSOR` positions to decide whether a tokenized `LINE` is still in use.
5. On each tokenization event, or via a small scheduled cleanup pass, it tries to detokenize older `LINE`s.
6. The cleanup rule is conservative:
   - walk oldest tokenized `LINE`s first
   - detokenize the first one that does not currently contain a `CURSOR`

This should let jsed keep SHALLOW_TOKENIZATION semantics over time instead of only at the moment of entry.

## Questions

- Should the canonical tokenization event come from `quickDescend(...)`, `tokenizeLine(...)`, or a higher-level `EditManager` orchestration point?
- Is "contains an active `CURSOR`" the only stop condition, or do we also need to avoid detokenizing the focused `LINE` in view mode?
- Should detokenization run immediately on each tokenization event, or be coalesced/scheduled to avoid churn during rapid navigation?
- What is the smallest useful detokenization unit: whole `LINE`, `LINE_SEGMENT`, or something else? The backlog item currently assumes whole `LINE`.

## Initial Proposal

- detokenize on the fly to reduce tokens
  - we create a Detokenizer object, inject it into EditManager
  - it listens for tokenization events
  - maybe it also needs to track cursors
    - for normal editing, there is one cursor which will get created and destroyed
    - but in future there could be remote cursors
    - if it has the current cursors, it can establish where in the DOM they are very quickly
  - it records what got tokenized
    - so it tracks calls to enterEditing / quickDescend
  - it wakes up, maybe on each tokenization event, and schedules clean up algorithm
  - clean up algorithm
    - go through oldest tokenized LINE's
    - detokenize first one where CURSOR is not present
