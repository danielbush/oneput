# feat: hitting enter splits paragraph

## Problem

There is no way to split a paragraph (or other LINE) into two while editing. The user needs to be able to press enter to create a new paragraph at the CURSOR position.

## Design

Split behaviour depends on CURSOR_STATE:
- Default: split before the current TOKEN (`splitBefore`)
- If CURSOR is in CURSOR_APPEND or CURSOR_INSERT_AFTER state: split after the current TOKEN (`splitAfter`)
- After splitting, CURSOR should sit on the new paragraph's first TOKEN

### Splitting inside INLINE's

When the CURSOR is inside an INLINE (e.g. `<em>`), splitting should go all the way up to the first NON_INLINE ancestor (the LINE). The CURSOR ends up at the beginning of the second LINE.

Consider `<p>aaa <em>bbb ccc ddd</em> eee</p>` with CURSOR on `ccc`:

**Split at mid-INLINE (CURSOR on `ccc`, not the first TOKEN in `<em>`):**
```
Before: <p>aaa <em>bbb ccc ddd</em> eee</p>
After:  <p>aaa <em>bbb</em></p>
        <p><em>ccc ddd</em> eee</p>
        CURSOR on `ccc`
```
The INLINE is duplicated across both LINE's, each retaining its portion of TOKEN's.

**Split at start of INLINE (CURSOR on `bbb`, the first TOKEN in `<em>`):**
```
Before: <p>aaa <em>bbb ccc ddd</em> eee</p>
After:  <p>aaa</p>
        <p><em>bbb ccc ddd</em> eee</p>
        CURSOR on `bbb`
```
The `<em>` stays intact in the second LINE. The first LINE gets only the TOKEN's before the INLINE.

If instead we only split the immediate parent (`<em>`), we'd get:
```
Before: <p>aaa <em>bbb ccc ddd</em> eee</p>
After:  <p>aaa <em>bbb</em> <em>ccc ddd</em> eee</p>
```
This is weird — the `<em>` is split but no new paragraph is created, which contradicts what "enter" means. So splitting up to the LINE is the right behaviour.

- TRANSPARENT_BLOCK's should be treated like INLINE_FLOW's.
- OPAQUE_BLOCK's should be treated the same eg p-tag, div-tag etc

## Initial Proposal

- CURSOR should split before the TOKEN by default
- if CURSOR is toggled to the "after token" position and/or we've typed a space after the token, then split after the current TOKEN
- CURSOR should sit on the new paragraph
- what happens if we're in a div? - do we repeat?
- what happens if we're in an em? - do we repeat?
- make sure we do some exploratory testing using test_doc.html
- `splitBefore`/`splitAfter` currently call `getLine(token)` internally to find the split ceiling. With TRANSPARENT_BLOCK, the passed-in LINE (from the CURSOR) may differ from the `getLine` result. For now these functions stay as-is (split relative to the nearest LINE), but when implementing enter-to-split we need to decide: should the split ceiling be the TRANSPARENT_BLOCK parent or the outer LINE? Likely the nearest LINE is correct (you split the immediate container), but verify with nested `<div>` structures
- Include PADDED_TOKEN testing for `splitBefore`/`splitAfter` — deferred from CURSOR_WALKS_NON_TOKENS__WORK housekeeping. Verify correct behaviour when splitting a PADDED_TOKEN or when a split produces a TOKEN adjacent to an ISLAND
