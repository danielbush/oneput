# discussion: refactoring `EditManager.handleInputChange(...)`

The backlog item to refactor `EditManager.handleInputChange(...)` no longer feels like ready implementation work.

We tried pushing more work into an `applyInputIntent(...)` function alongside `decideInputIntent(...)`, but the result was not satisfying. The split blurred the boundary between orchestration and mutation, and the tests for the extracted function became harder to read than the current whole-flow tests.

## Current concern

The original goal was straightforward:

- make `handleInputChange(...)` much smaller
- keep intent calculation explicit
- make the logic easier to test

But the attempted extraction exposed a tension:

- some manipulation still wanted to stay in `handleInputChange(...)`
- some manipulation moved into `applyInputIntent(...)`
- the resulting split did not make responsibilities clearer

That suggests the seam is not obvious yet.

## Testing concern

The current `EditManager.handleInputChange(...)` tests use symbolic notation to describe the whole editor transition. That feels valuable because the behavior is fundamentally end-to-end at the `EditManager` layer.

Moving behavior into `applyInputIntent(...)` pushed toward narrower tests with nulled dependencies, but those tests felt harder to read and less representative of the real editing flow.

Current lean:

- keep the symbolic-notation tests that represent the whole thing
- avoid extracting logic merely to create a narrower unit seam
- only split if the resulting modules and tests are actually clearer

## Questions

- Is `applyInputIntent(...)` the wrong seam?
- Should `handleInputChange(...)` stay mostly intact, with only smaller semantic helpers extracted under it?
- If we do extract something, should it be a pure decision helper, a semantic action layer, or a command object rather than an "apply intent" function?
- What test seam preserves the readable symbolic-notation examples while still allowing some lower-level logic to be tested directly?

- this discussion is specifically about whether `handleInputChange(...)` should be split further, and if so where
