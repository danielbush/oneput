# feat: selections

Created from the `selections` item in `work/backlog/jsed.md` under `## Critical path`.

## Motivations

Selection should behave like text selection in a text editor or word processor: the user holds a modifier (shift) and moves the CURSOR, growing a range that extends over multiple LINE_SIBLING's in the move direction. A range can then be typed over, deleted, or collapsed.

## Design sketch (as of 2026-04-15)

Selection is an extension of editing mode, managed by a new `TokenSelection` class seeded from the current TOKEN as both anchor and head.

**Visual mechanism — SELECTION_WRAPPER:** rather than marking individual TOKEN's with a class (which leaves the whitespace between them unstyled and looks broken), the selection is represented by one or more `SELECTION_WRAPPER` elements — inline spans inserted into the DOM as the direct parent of a contiguous run of LINE_SIBLING's that share a DOM parent. Whitespace text nodes between TOKEN's live inside the wrapper, so the background paints as a continuous stripe.

Because the wrapper is inserted *between* an ancestor and its TOKEN's (not around the ancestor), CSS inheritance and `text-decoration` flow through unchanged — italic from `<em>`, bold from `<strong>`, underline from `<u>` or inline styles all still apply to the wrapped TOKEN's. Verified manually with `<p>This is <u><span>just a</span> simple</u> ...</p>` and the inline-style equivalent.

**Parent boundaries:** entering an `<em>` (or any INLINE_FLOW) means a new wrapper starts inside the `<em>`, still as direct parent of the TOKEN's there — preserving the styling chain. Leaving the `<em>` starts another wrapper back in the outer parent. The `<em>`'s tag markup sits unwrapped between adjacent wrappers; the selection visually "skips" the tag, which is acceptable. Rule: **never use a sibling-of-em wrapper approach** — always direct-parent-of-TOKEN's — so the styling chain is preserved.

`TokenSelection` state: `anchor`, `head`, `wrappers: SELECTION_WRAPPER[]` (ordered, one per contiguous same-parent run).

**Collapse** unwraps each wrapper in place, leaving TOKEN's and whitespace text nodes untouched.

**Scope for first pass:** within a single LINE (tasks 1 and 2). Cross-LINE selection (task 4) deferred. Typing-to-replace and delete over a range are also deferred until the visual growth mechanism is working.

**Standalone phase:** `TokenSelection` currently does its own LINE_SIBLING traversal via `sibwalk.getNextLineSibling` / `getPreviousLineSibling` and does not touch `TokenCursor`. The CURSOR stays where it was when the selection started, and the wrapper grows out ahead of it. Once we can see this render, we'll decide whether the CURSOR should follow the head (option b: one source of truth) or stay pinned at the anchor (option a: cleaner separation).

## Progress (as of 2026-04-15)

Stubs in place — pressing the key bindings flows through the stack but currently just advances the head pointer (console logs for now in TokenSelection); no DOM mutation yet.

- `packages/jsed/src/TokenSelection.ts` — new class. Constructor seeds `anchor`/`head` from a TOKEN. `extendNext` / `extendPrevious` walk LINE_SIBLING's via `sibwalk`. `collapse()` placeholder. All wrapper DOM mutation left as TODOs in the method bodies.

- `apps/jsed-demo/src/lib/oneput/app/EditDocument.ts` — new `EXTEND_NEXT` / `EXTEND_PREVIOUS` actions bound to `Shift+$mod+l` / `Shift+ArrowRight` and `Shift+$mod+h` / `Shift+ArrowLeft`. Call into the EditManager stubs.

2. Add the CSS for `.jsed-selection` (background-only, styling-neutral).
3. Handle the parent-boundary case — start a new wrapper when the next LINE_SIBLING is in a different immediate parent (entering/leaving an INLINE_FLOW).
