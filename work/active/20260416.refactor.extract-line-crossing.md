# refactor: extract line-crossing out of TokenCursor

## Motivation

`TokenCursor` currently does more than manage a cursor within a LINE. When
`moveNext` / `movePrevious` runs out of LINE_SIBLING's it falls into
`findNextLineTarget` / `findPreviousLineTarget`, which walk past the current
LINE using the document root as ceiling, call `quickDescend` on the new LINE
(shallow tokenization on arrival), and for the backward case use
`findLastCursorTarget` to resolve the deepest reachable TOKEN in the previous
LINE. This pulls tokenization + document-level walking into a class whose
purpose is "manage the CURSOR within a LINE".

The original design intent was different: the cursor would exhaust the LINE
and then EditManager (or some coordinator) would be responsible for promoting
it to the next LINE — tokenizing on arrival and re-seating the cursor. That
separation is cleaner, and the selections work just made it pressing:
`TokenSelection` needs the same cross-LINE capability for task #4 of
`20260414.feat.selections.md`, and duplicating the current TokenCursor logic
(or giving TokenSelection its own `Tokenizer` dependency) would multiply the
problem.

The right move is to push the cross-LINE responsibility UP into a coordinator
where it belongs, and shrink TokenCursor back to its intra-LINE core. Then
TokenSelection can stay intra-LINE too, and both participate in the same
coordinator-driven line-crossing flow.

## Prerequisite: lock in Tokenizer and Detokenizer behavior

Before touching TokenCursor, we need unit tests for `Tokenizer` and the
detokenize path (`detokenizeLine` / background cleanup in `Tokenizer`).
They were built + exercised via exploratory testing only. We're about to do
surgery adjacent to them, and the coordinator will be a new consumer — we
need a safety net before we start moving code.

The tests should cover:

- `Tokenizer.quickDescend` on a fresh LINE produces expected TOKEN's
- `quickDescend` is a no-op on an already-tokenized LINE
- `detokenizeLine` removes TOKEN wrappers and normalizes text nodes back
- Background cleanup detokenizes the oldest LINE and skips the CURSOR LINE
- Any error paths we care about (Result shape)

Level: probably `Tokenizer` directly (intermediate), since this is the seam
we're about to pivot on. Flag to jsed testing conventions: this is one of
those cases the CLAUDE.md advice calls out as "stop and ask" — doing this at
EditManager level would be awkward because the behavior is too internal. The
testing target here is specifically "lock the current behavior before
refactoring", which is a legitimate reason to test at the class level.

## Design sketch

**`exhausted` signal shape**

Use a discriminated return value, not an event and not a `Result`.
Exhausting a LINE is not an error; it is normal control flow that the
coordinator (`EditManager`) should handle.

- `TokenCursor.moveNext()` / `movePrevious()` should return a small outcome
  union such as:
  - `{ type: 'moved' }`
  - `{ type: 'stayed' }` — used for intra-LINE state changes that do not
    actually move, e.g. cancelling `CURSOR_INSERT_BEFORE` /
    `CURSOR_INSERT_AFTER`
  - `{ type: 'exhausted' }`
- `TokenSelection.extendNext()` / `extendPrevious()` can likely use the
  simpler pair:
  - `{ type: 'moved' }`
  - `{ type: 'exhausted' }`

Important meaning: `exhausted` means "exhausted this LINE", not "there is
definitely another LINE to promote into". The intra-LINE class reports that it
cannot continue locally; `EditManager` then decides whether a neighbour LINE
exists and performs promotion if it does.

**Intra-LINE classes**

- `TokenCursor.moveNext()` / `movePrevious()` stop calling
  `findNextLineTarget` / `findPreviousLineTarget`. They either move within
  the current LINE or return an `exhausted` outcome via the discriminated
  return value above.
- `TokenSelection.extendNext()` / `extendPrevious()` do the same.

Both classes lose their cross-LINE branches, their dependency on
`findNextLineTarget` / `findPreviousLineTarget` / `findLastCursorTarget`, and
any implicit reliance on `document.root` as ceiling. They become smaller and
testable without a `Tokenizer`.

**Line-crossing coordinator**

Owns the tokenizer and the document root. Holds the line-crossing helpers
currently private to TokenCursor:

- `findNextLineTarget(from: HTMLElement)` — walk past `from`'s LINE to the
  first LINE_SIBLING in the next LINE, `quickDescend` on arrival, return the
  first TOKEN.
- `findPreviousLineTarget(from: HTMLElement)` — mirror, returning the last
  reachable TOKEN of the previous LINE via a public version of
  `findLastCursorTarget`.

When a cursor or selection reports `exhausted`, the coordinator:

- for **cursor**: calls `findNextLineTarget`/`findPreviousLineTarget`, then
  `cursor.setToken(newTarget)`. No state reconciliation.
- for **selection**: calls the same helper to locate the target in the
  neighbour LINE, then calls a new method on TokenSelection — working name
  `extendIntoNewRun(target)` — which opens a fresh SELECTION_WRAPPER around
  `target` in its own parent, pushes it onto `wrappers`, and sets
  `head = target`. Existing wrappers in the prior LINE are untouched. The
  `wrappers: HTMLElement[]` array already documents itself as
  "ordered, one per contiguous same-parent run" — this just extends that
  invariant naturally across LINE boundaries.

Shrinking back across a LINE boundary requires no new code:
`shrinkBack`/`shrinkFront` already check `back.contains(prev)` — when head
retreats out of the neighbour LINE's wrapper, the back wrapper doesn't
contain the new `prev`, so it unwraps and pops, and head lands back inside
the prior LINE's existing last wrapper. Verified by the single-LINE em-crossing
tests (`TokenSelection.test.ts`) which exercise exactly this case within one
LINE.

**Where does the coordinator live?**

**Decision: fold into EditManager.** The three helpers (`findNextLineTarget`,
`findPreviousLineTarget`, `findLastCursorTarget`) are ~70 lines total — too
small to justify a new module. EditManager already owns the Tokenizer,
TokenCursor, TokenSelection, and document, so a separate `LineCrosser` would
just receive them as constructor args and act as pass-through indirection.
The call sites (`handleLeft`/`handleRight`/`extendNext`/`extendPrevious`) are
already in EditManager, so the `if (exhausted) { promote() }` pattern reads
naturally inline. Deep-modules applies to making TokenCursor/TokenSelection
simpler by extracting cross-LINE concern *up* — but "up" stops at EditManager.
If the crossing logic grows significantly later (cross-LINE undo/delete),
extracting to a separate module is trivial since the methods will be grouped.
