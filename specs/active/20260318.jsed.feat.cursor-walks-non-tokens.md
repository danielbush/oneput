The CURSOR currently visits TOKEN's and recurses into INLINE's.
We want to expand this behaviour so that LINE_SIBLING's include other elements besides TOKEN's.

We need to review the taxonomy.

By definition FOCUSABLE means visit=yes for FOCUS .

- (1) non-INLINE FOCUSABLE where...
  - FOCUS visit=yes, descend=no
  - CURSOR visit=yes, descend=no
  - COMMENT: let's assume these are ISLAND's as we defined before except the CURSOR can now visit; this would be something that naturally sits on the line alongside tokens but should be externally managed such as katex equations that form part of the sentence or even in latex block mode; this is the classic ISLAND (TODO: verify definition fits pending the change to CURSOR now visiting it)
- (2) non-INLINE FOCUSABLE where...
  - FOCUS visit=yes, descend=yes
  - CURSOR visit=yes, descend=yes
  - COMMENT: I think ISLAND's (1) are special, and we should distinguish them by a class or they have obvious signatures such as katex; this category might be a default; that is it's not INLINE, it's not an ISLAND, so the default behaviour is to let the CURSOR descend into it like an INLINE
- (3) non-INLINE FOCUSABLE where...
  - FOCUS visit=yes, descend=yes
  - CURSOR visit=yes, descend=no
  - COMMENT: because we have (2), we want to allow the case where for some reason we don't want the CURSOR to descend; because FOCUS can descend them, I tend to think of these as NESTED_LINE's; I don't think these should be common, so these might have some kind of class in the DOM to identify them;  so a user would have to mark them as being non-descendable by the CURSOR;
  - TODO: I'm not sure I like NESTED_LINE - happy to rethink the name, and also adjust the vocab definitions accordingly
- (4) non-INLINE FOCUSABLE where...
  - FOCUS visit=yes, descend=yes
  - CURSOR visit=no, descend=no
  - COMMENT: finally because we have (3) it opens the possibility for (4) where we have something that is part of the document content but we just don't want the cursor to visit it at all; so it is like an IGNORABLE but only for the CURSOR; possibly these could be markers that are part of the document but are never edited via the cursor; but I'm struggling for a use case
- (5) TODO: do other combinations of FOCUSABLE's make any sense?

Key decision for discussion: how to distinguish (1) - (4)?  As mentioned int he COMMENT's above, I think ISLAND's (1) are special and are flagged in some way including an epxlicity class that we can add; (2) is a default, so this means if the user doesn't do anything, the CURSOR will descned them and visit all their TOKEN's just like an INLINE; and (3) and (4) are special cases that we have to explicitly flag with a CSS class.

The next thing to be wary of is getLine and how the current LINE is established especially during repeated calls to getNextLineSibling/getPreviousLineSibling as used in moveNext, movePrevious of TokenCursor (the CURSOR).  The CURSOR should never explicitly be set on (4) and should probably refuse if asked (setToken); when finding next or previous LINE_SIBLING's it should obviously not visit (4) - getLine should never treat it as a LINE. If T is (3), the system knows it's a "cursor ISLAND" because it's explicitly marked, so it can never be a LINE and getLine should be adjusted accordingly.  If T is (2), it treats it like INLINE, but the problem here is that it looks like and could be a LINE in its own right.  Perhaps getParentLine is the solution for all cases?  It gets the parent element, and the calls getLine on it.  That would mean getNextLineSibling / getPreviousLineSibling always look to the parent to establish the LINE.
