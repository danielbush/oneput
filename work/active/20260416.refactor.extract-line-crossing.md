# refactor: consolidate navigation into TokenCursor

> **Direction change (2026-04-17):** originally framed as *extract* line-crossing UP into EditManager. Three extraction commits later, the result was logic smeared across `TokenCursor`, `EditManager`, and `quickDescend` with no clear owner. New direction: pull cross-LINE walking + on-arrival tokenization DOWN into `TokenCursor`. `TokenCursor` already takes `Tokenizer` and tokenize-on-move is naturally cursor work. `EditManager` shrinks to pure mode-switching and dispatch. The original "extract" framing of this ticket is preserved below in git history; the rewrite reflects the new direction.

## Motivation

`TokenCursor` already takes `Tokenizer` and already drives `quickDescend`. Cross-LINE movement + tokenize-on-arrival is the same job: "advance the cursor to the next reachable TOKEN, tokenizing the new LINE on arrival". Splitting that job between `TokenCursor` (intra-LINE) and `EditManager` (cross-LINE) — the original plan — fragmented a single concern across two layers and a free-floating `quickDescend` helper.

Consolidating into `TokenCursor` also dissolves the `exhausted` signal (only needed because the work was split) and gives `TokenSelection` a clean reuse path: spin up a second `TokenCursor` for the head and let it walk however far it needs to, with the editing cursor pinned at the anchor. That closes the open "CURSOR follows SELECTION head vs pinned at anchor" question on `20260414.feat.selections.md` in the **pinned** direction.

## Design

**`TokenCursor` absorbs:**

- cross-LINE walking (`findNextLineTarget`, `findPreviousLineTarget`, `findLastCursorTarget` — the helpers currently mid-extraction)
- on-arrival `quickDescend` for any LINE it walks into
- the `enterEditing`-style "find first reachable target in this LINE" entry point when seeded from a FOCUSABLE

**`Tokenizer` keeps:** the tokenize/detokenize service plus background cleanup orchestration that skips the active CURSOR_LINE. Unchanged role.

**`EditManager` becomes:** mode switching (view ↔ editing), dispatching key actions, routing REQUEST_FOCUS, routing UserInput change/selection-change. No cross-LINE coordination, no `exhausted` handling.

**`TokenSelection`:** owns its own internal `TokenCursor` instance for the head. Editing cursor stays at the anchor. Cross-LINE extension uses the head-cursor's own cross-LINE walk; the existing `wrappers` invariant ("ordered, one per contiguous same-parent run") extends across LINE boundaries naturally. A new `extendIntoNewRun(target)` opens a fresh SELECTION_WRAPPER when the head-cursor lands in a new parent.

**Vocabulary:** unchanged. CURSOR-as-concept still describes LINE_SIBLING navigation within a LINE; the code object simply handles the seam between LINEs internally.

## What's already landed (keep)

## Approach: move forward, not revert

The three extraction commits (`6da36de1`, `b43584b6`, `2285699d`) stand. We pull the responsibility back into `TokenCursor` by going forward — re-absorbing the cross-LINE logic into a unified shape rather than reverting and starting over.

The hard step is "disperse" `quickDescend`'s logic into `TokenCursor` so cross-LINE moves can tokenize-on-arrival without going through the standalone helper. We get there in small, ordered steps so each one stands on its own.
