## Changes

- **CURSOR_STATE vocabulary** — defined five states (CURSOR_OVERWRITE, CURSOR_APPEND, CURSOR_PREPEND, CURSOR_INSERT_AFTER, CURSOR_INSERT_BEFORE) in `docs/vocabulary.md` describing how the user cycles through editing modes on a TOKEN
- **Renamed constants** — `TOKEN_APPEND_CLASS` → `CURSOR_APPEND_CLASS`, etc. to align with the vocabulary. CSS class string values remain unchanged (a follow-up spec exists to rename those too: `work/active/rename-cursor-css-classes.md`)
- **TokenCursor tests** — 24 tests covering motion through simple LINE's, single/nested/adjacent INLINE_FLOW's, and all five CURSOR_STATE transitions including the cancel-on-reverse-motion behavior
- **TokenCursorBase extraction** — new base class holding TOKEN reference, JSED_CURSOR_CLASS management, and protected focus-class methods. TokenCursor extends it with motion, editing, and CURSOR_STATE logic
- **CursorMarkers deleted** — its ~30 lines of marker logic folded directly into TokenCursor. The round-trip through `ITokenCursor` (calling `addFocusClasses`/`removeFocusClasses` back on itself) replaced by direct protected method calls
- **ITokenCursor cleaned up** — removed `addFocusClasses`/`removeFocusClasses` from the public interface (no external callers existed)

## Decisions

- **TokenCursor is IN_MEMORY, not OUTSIDE_WORLD** — nullables-refactor analysis confirmed no DUAL_FACTORY needed. All dependencies (DOM, TokenManager, CursorMarkers) are in-memory. Tests use real objects with no mocks, which is the nullables philosophy applied to IN_MEMORY code
- **Two-layer inheritance, not three** — considered splitting motion from editing into separate layers, but editing operations also call `setTokenInternal` and interact with the same token state. An artificial seam would make code harder to follow. One base (plumbing) + one subclass (behavior) is the right depth
- **CursorMarkers absorbed rather than kept as composition** — CursorMarkers existed because focus-class management was public. With it protected on the base, the marker logic naturally lives in TokenCursor alongside the motion methods that check `isInsertingBefore`/`isInsertingAfter`. No separate object needed
- **`getNextLineSibling`/`getPreviousLineSibling` only consumed by TokenCursor** — confirmed via jcodemunch + grep. All three production call sites are in TokenCursor (moveNext, movePrevious, toggleCollapsePrevious)
