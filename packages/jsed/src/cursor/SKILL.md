This is a skill file that is intended to be used for code in the same directory. You can invoke it with the local-lens skill.

- This directory is one subsystem of the jsed codebase. The same testing shape should apply to other subsystems such as lib/editor: exhaustive detail belongs in lower-level load-bearing modules first, while the higher-level layer gets focused integration coverage.
- For each method in Cursor class in Cursor.ts or the one specified by the user
  - identify which lower-level operations are used; these are the load-bearing operations
    - in this directory, these will usually be in lib/ops and lib/core
  - identify any operations outside lib/{cursor,ops,core}; how many? how heavily are they used
  - look at how well tested each load-bearing operation is and identify any missing cases for these operations and write tests for them in the lower-level module that owns the behavior
    - prioritize detailed coverage here first; lower-level tests are cheaper to understand and better at documenting edge cases
  - if the method mutates the DOM, and implements UndoRecord with a static run method...
    - check for the following
      - tests for the static run method of the class
        - don't be exhaustive, rely on the exhaustive testing of the load bearing operations in lib/ops, lib/core
        - focus on testing key happy and sad paths
        - focus on testing for integration issues
      - undo tests
      - redo tests
  - the higher-level method should also be tested but focus on
    - obvious happy/sad paths and test integration
    - detailed tests at this level are acceptable when they catch integration behaviour that lower-level tests cannot
    - prefer moving exhaustive detail to load-bearing operations first, then add only enough higher-level coverage to show the pieces work together
    - integration
      - such as the management of multiple undo/redo actions after multiple operations.
