# feat: REC_NEXT/SIB_NEXT and REC_PREV/SIB_PREV close CURSOR and move FOCUS

## Problem

When the CURSOR is open (editing a LINE), pressing REC_NEXT/SIB_NEXT or REC_PREV/SIB_PREV does nothing useful. The user has to explicitly exit edit mode before navigating. This adds friction to the editing flow.

## Design

The user is in edit mode (CURSOR active on a TOKEN within a LINE). Pressing a navigation key (REC_NEXT, SIB_NEXT, REC_PREV, SIB_PREV) should exit edit mode entirely and return to view mode — browsing the document structure via FOCUS without any active CURSOR:

1. Close the CURSOR (de-tokenize, clean up edit state)
2. The user is now back in view mode — no TOKEN is selected, no editing is happening
3. Move FOCUS to the appropriate FOCUSABLE using the corresponding Nav action

This ties into the "focus twice to edit" model (see FOCUS_TWICE_TO_EDIT__DISCUSSION): navigation keys take you back to viewing, and you'd need to focus the same element twice again to re-enter edit mode.

## Implementation

In `apps/jsed-demo`, navigation is split across two AppObject's:
- **ViewDocument** — has REC_NEXT, SIB_NEXT, etc. bindings that move FOCUS via Nav
- **EditDocument** — has TOKEN-level editing bindings but no navigation keys

EditDocument needs its own versions of REC_NEXT, SIB_NEXT, REC_PREV, SIB_PREV that:
1. Close the CURSOR (exit edit mode)
2. Move FOCUS via Nav (same as ViewDocument)
3. Transition back to ViewDocument (the user is now in view mode)

These mirror ViewDocument's nav bindings but add the edit-mode teardown and AppObject transition.
