# discussion: allow CURSOR to visit CURSOR_TRANSPARENT's

Currently the CURSOR passes through CURSOR_TRANSPARENT elements (INLINE_FLOW, TRANSPARENT_BLOCK) without stopping on them — it only visits their TOKEN children. What if the CURSOR could also visit these elements themselves? CURSOR_TRANSPARENT would become a toggle rather than a fixed behaviour.

## Positives

- When the CURSOR lands on an INLINE_FLOW like `<em>`, we could offer CURSOR operations that act on the INLINE_FLOW itself — e.g. extending, shrinking, or removing the `<em>` wrapper — without needing a separate mechanism to access it
- Unifies structural editing with text editing under the same CURSOR navigation model

## Negatives

- Users have to navigate sub-LINE structures (INLINE_FLOW's, TRANSPARENT_BLOCK's) in addition to TOKEN's — more stops per line, potentially tedious
- Deeply nested structures (e.g. `<em>` inside `<strong>` inside `<a>`) would multiply the number of CURSOR stops, making TOKEN-to-TOKEN navigation harder
- May need a way to toggle between "visit everything" and "visit TOKEN's only" modes to keep navigation ergonomic

## Current thinking (26-Mar-2026)

Leaning towards **not** visiting INLINE_FLOW's or similar. The main argument: it adds an extra stop the user has to navigate through to reach the next TOKEN, which makes the common case worse. The structural editing use case (modifying the `<em>` wrapper etc.) doesn't require CURSOR visitation — FOCUS is already set automatically on the parent element, so we can surface a context menu in Oneput to modify the element under FOCUS instead. This keeps CURSOR navigation tight (TOKEN-to-TOKEN) and handles structural editing through a separate, more appropriate mechanism.

## Open questions

- If structural editing goes through FOCUS + context menu, does this discussion reduce to "no change needed"?
- How does the FOCUS-based context menu interact with OPAQUE_BLOCK? When the user enters an OPAQUE_BLOCK, the FOCUS moves to it — the context menu could offer "make transparent" or structural operations at that level too.
