## Proposal

The CURSOR currently visits only TOKEN's (and recurses through INLINE's). We expand it so that LINE_SIBLING's include three categories of non-TOKEN FOCUSABLE's:

| Cat | visit | descend | Detection | Examples |
|-----|-------|---------|-----------|----------|
| ✅ (1) ISLAND | CURSOR=yes, FOCUS=yes | CURSOR=no, FOCUS=no | Implicit (e.g. `.katex`, `<img>`) or explicit class | Katex equations, images — opaque elements the CURSOR hops over |
| (2) BLOCK_TRANSPARENT | CURSOR=no, FOCUS=yes | CURSOR=yes, FOCUS=yes | Default for any non-INLINE, non-ISLAND FOCUSABLE | Nested `<div>`, `<section>`, inline-block `<span>` — CURSOR descends into their TOKEN's seamlessly |
| (3) CURSOR_BOUNDARY | CURSOR=yes, FOCUS=yes | CURSOR=no, FOCUS=yes | Explicit `jsed-cursor-opaque` class | Elements FOCUS can descend but CURSOR treats as opaque |

Category (4) (CURSOR visit=no, descend=no) is deferred — no concrete use case yet.

### Taxonomy summary

Non-TOKEN FOCUSABLE's group into two CURSOR behaviors:

- **CURSOR_TRANSPARENT** (descend=yes, visit=no) — the CURSOR passes through to visit TOKEN children
  - **INLINE** — inline-level markup (`<em>`, `<a>`)
  - **BLOCK_TRANSPARENT** — everything else: block, inline-block, etc. (nested `<div>`, `<section>`)
- **CURSOR_OPAQUE** (descend=no, visit=yes) — the CURSOR lands on the element itself
  - **ISLAND** — externally managed content (katex, `<img>`)
  - **CURSOR_BOUNDARY** — explicitly marked elements FOCUS can descend but CURSOR cannot

### Key design decisions

**LINE as explicit parameter.** `TokenCursorBase.setToken` calculates the LINE when assigned a TOKEN. `getNextLineSibling(el, line)` and `getPreviousLineSibling(el, line)` take the LINE as an explicit ceiling — they never call `getLine` internally. This solves the deep nesting problem: the LINE is determined once at entry time, not inferred per-hop.

**`getLine` stays as-is** for FOCUS-level code (Nav). It just stops being used by CURSOR-level code.

**visit/descend predicates expand:**
- `visit`: `isToken || isIsland || isCursorBoundary` — things the CURSOR lands on
- `descend`: `isInline || isBlockTransparent` — things the CURSOR passes through transparently (visiting their TOKEN children)

**Naming.** NESTED_LINE remains a structural term (a LINE inside another LINE). The behavioral annotation for (3) is `CURSOR_BOUNDARY` / `jsed-cursor-opaque`.

## Plan for (2) BLOCK_TRANSPARENT

**Goal:** The CURSOR descends into BLOCK_TRANSPARENT elements seamlessly (like INLINE's), visiting their TOKEN children as if they were part of the parent LINE. 18 failing tests already written.

### The core problem: LINE ceiling

Currently `getNextLineSibling(el)` and `getPreviousLineSibling(el)` call `getLine(el)` internally to find the ceiling for the walk. `getLine` walks up to the nearest FOCUSABLE that's not a TOKEN, INLINE, or ISLAND — which is `isLine`. But a BLOCK_TRANSPARENT element (e.g. nested `<div>`) IS a LINE by that definition. So for a TOKEN inside `<div id="inner">` inside `<div id="outer">`, `getLine(token)` returns `#inner`, and the walk never escapes to `#outer`.

### Approach: LINE as explicit parameter

The LINE is determined once — when the user enters edit mode on a FOCUSABLE — and passed explicitly to all CURSOR-level traversal. `getLine` stays as-is for FOCUS-level code (Nav); it just stops being used by CURSOR-level code.

### Steps

#### Step 1: Add `isBlockTransparent` to traversal.ts

Sits alongside `isInline`, `isLine`, and `isIsland` — all element classification predicates now live in traversal.ts. BLOCK_TRANSPARENT only matters for CURSOR traversal.

```typescript
/** BLOCK_TRANSPARENT — the default for any non-INLINE, non-ISLAND FOCUSABLE.
 *  The CURSOR descends into it seamlessly (like INLINE). */
export function isBlockTransparent(el: Node | ChildNode | ParentNode | null): boolean {
  return isLine(el) && !isCursorBoundary(el);
}
```

For now `isCursorBoundary` doesn't exist yet (scenario 3), so this is just `isLine(el)`. Add a stub `isCursorBoundary` that returns `false` so the intent is clear and (3) has a hook point.

#### Step 2: Change LINE_SIBLING traversal signatures (in traversal.ts)

Change `getNextLineSibling` and `getPreviousLineSibling` to take `line` as an explicit parameter:

```typescript
export function getNextLineSibling(el: HTMLElement, line: HTMLElement): HTMLElement | null
export function getPreviousLineSibling(el: HTMLElement, line: HTMLElement): HTMLElement | null
```

Remove the internal `getLine(el)` call. The `line` parameter becomes the walk ceiling.

Update `descend` predicate from `isInline` to:
```typescript
descend: (n) => isInline(n) || isBlockTransparent(n)
```

Update `getFirstLineSibling` similarly:
```typescript
descend: (n) => n === line || isInline(n) || isBlockTransparent(n)
```

`isLineSibling` stays unchanged — `isToken || isIsland`. BLOCK_TRANSPARENT is descended through, not visited.

Update the re-exports in token.ts to include `isBlockTransparent` so `import * as token` consumers can access it without changing their imports.

#### Step 3: TokenCursorBase stores and passes the LINE

`TokenCursorBase.setToken` already calculates the LINE (`getLine`). Store it and expose it. But the LINE used for CURSOR traversal should be the one determined at entry time, not recalculated per-hop.

- Add a `line` field to `TokenCursorBase`, set when the CURSOR is first positioned
- `moveNext` / `movePrevious` pass `this.line` to `getNextLineSibling` / `getPreviousLineSibling`
- All other callers of `getNextLineSibling` / `getPreviousLineSibling` updated to pass the LINE

#### Step 4: TokenManager tokenizes BLOCK_TRANSPARENT lazily on cursor move

Currently `tokenizeLine` does SHALLOW_TOKENIZATION — one LINE at a time. For BLOCK_TRANSPARENT to work, the TOKEN's inside nested blocks need to exist before the CURSOR can walk to them.

~~Rejected approach: `tokenizeDeep` — tokenize the focused FOCUSABLE and all BLOCK_TRANSPARENT descendants eagerly at entry time. Problem: if the user focuses a high-level container (or the root), this tokenizes the entire document, defeating SHALLOW_TOKENIZATION. Unacceptable for large documents.~~

**Approach: lazy tokenization on cursor move.** When `moveNext`/`movePrevious` walks into a BLOCK_TRANSPARENT that hasn't been tokenized yet, tokenize just that one BLOCK_TRANSPARENT before descending. This bounds the cost to one level per cursor move and preserves the on-demand spirit of SHALLOW_TOKENIZATION.

Concretely:
- `moveNext` / `movePrevious` already call `getNextLineSibling` / `getPreviousLineSibling`. After step 2, the traversal descends into BLOCK_TRANSPARENT's. If a BLOCK_TRANSPARENT has no TOKEN's yet, the walk passes through it (finding nothing) and continues.
- To fix: when the walk enters a BLOCK_TRANSPARENT, check if it's been tokenized; if not, tokenize it on-the-fly and restart/continue the walk from there.
- This can be done in `TokenCursor.moveNext`/`movePrevious` (check if the result is null but untokenized BLOCK_TRANSPARENT's exist ahead) or inside the traversal walk itself (tokenize-on-enter callback).
- `TokenManager.tokenize` already handles single-LINE tokenization — reuse it for each BLOCK_TRANSPARENT encountered.

#### Step 5: Update callers

- `TokenCursor.moveNext` / `movePrevious` — pass `this.line` (these call `token.getNextLineSibling` / `token.getPreviousLineSibling` via the re-export)
- `TokenCursor` editing ops (`splitBefore`, `splitAfter`, `joinNext`, `joinPrevious`, `toggleCollapseNext`, `toggleCollapsePrevious`) — pass `this.line` where they call LINE_SIBLING traversal
- `EditManager` — no change needed (still calls `tokenize` for the focused LINE); lazy tokenization happens at cursor-move time
- `InputManager` — if it references LINE_SIBLING traversal, pass the LINE
- token.ts internal callers — `splitBefore`/`splitAfter` call `getLine(token)` for the ceiling; these should take `line` as a parameter instead (or callers pass it)
- Tests: `collectForward` / `collectBackward` helpers need updating to pass a LINE parameter

- 9 failing `lineSibling.test.ts` tests for (2) should pass
- 5 failing `TokenCursor.test.ts` tests for (2) should pass
- All (1) ISLAND tests still pass (no regression)
- (3) CURSOR_BOUNDARY tests still fail (expected — not implemented yet)

### Risks and edge cases

- **Lazy tokenization complexity** — the cursor walk needs to detect untokenized BLOCK_TRANSPARENT's and tokenize on-the-fly. This adds a side effect to what is currently a pure traversal. The cleanest approach is likely a callback or a wrapper in TokenCursor rather than baking tokenization into the walk itself.
- **`getLine` callers outside CURSOR code** — Nav and other FOCUS-level code still calls `getLine`, which returns the nearest LINE. This is correct: FOCUS navigates to the BLOCK_TRANSPARENT `<div>` as a FOCUSABLE, even though the CURSOR passes through it. No change needed.
- **ISLAND inside BLOCK_TRANSPARENT** — should work naturally: the walk descends into BLOCK_TRANSPARENT, then when it encounters an ISLAND, `isLineSibling` returns true (visit) and `descend` returns false. Already covered by existing (1) tests if we add a combined test case.
- **INLINE inside BLOCK_TRANSPARENT** — same: walk descends into BLOCK_TRANSPARENT, then descends through INLINE, visits TOKEN's inside. The nested `div` inside `<em>` test already covers this.

### Known issues (found in exploratory testing)

- **Space loss around ISLAND** — when `tokenizeLine` processes text adjacent to an ISLAND (e.g. `<p>aaa <span class="katex">x²</span> bbb</p>`), the leading space in `' bbb'` is stripped by `split(/\s+/).filter(Boolean)`. Normally the previous TOKEN's trailing space provides the visual gap, but the ISLAND has no trailing space — so the ISLAND and `bbb` render flush.
  - **Proposed approach: PADDED_TOKEN.** The TOKEN after the ISLAND owns the gap via a leading space. This introduces PADDED_TOKEN as the counterpart to COLLAPSED_TOKEN. A TOKEN has 2² = 4 spacing states:
    - unpadded + uncollapsed: `'foo '` — the default
    - unpadded + collapsed: `'foo'` — COLLAPSED_TOKEN (existing)
    - padded + uncollapsed: `' foo '` — PADDED_TOKEN
    - padded + collapsed: `' foo'` — PADDED_TOKEN + COLLAPSED_TOKEN
    PADDED_TOKEN adds a leading space; COLLAPSED_TOKEN removes the trailing space. Both are per-TOKEN, both are togglable by the user.
    - PADDED is needed when the previous LINE_SIBLING doesn't carry its own trailing space (e.g. an ISLAND).
    - In NEGATIVE_SPACE, a stale leading space (e.g. if the ISLAND is later removed) is visually harmless — the browser collapses it. A periodic or hook-based cleanup could normalize it.
    - In POSITIVE_SPACE, the leading space is visible and correct — the original whitespace should be preserved.
    - User toggling: "toggle pad" key binding when previous LINE_SIBLING is an ISLAND, mirroring "toggle collapse" for TOKEN-to-TOKEN gaps.
    - Implementation: `createToken`/`replaceText` currently normalize to `text + ' '`. They'd need to preserve a leading space if present. `getValue` currently trims — needs to be aware of leading space. Track via a CSS class (like `JSED_TOKEN_COLLAPSED` for trailing → e.g. `JSED_TOKEN_PADDED` for leading).
    - Wrapping the ISLAND was considered and rejected — ISLAND internals are externally managed (e.g. katex renderer), wrapping changes `parentNode` which could break re-renders/event handlers, and wrapping block-level elements (scenarios 2/3) in inline spans is invalid HTML.
- **Typing over ISLAND** — when the CURSOR lands on an ISLAND, `onTokenChange` fires with the ISLAND element. The input pipeline puts its text content (`x²`) into the input field, and the user can type over it — hitting `replace`/`replaceText` and corrupting the ISLAND's DOM. The `onTokenChange` callback (or the input layer above it) needs to distinguish TOKEN from non-TOKEN LINE_SIBLING's so it can make the input read-only or disable it while the CURSOR is on an ISLAND.
