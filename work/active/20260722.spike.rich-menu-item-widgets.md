# spike: rich menu-item widgets (alternative to MenuLike)

Alternative to [MENU_LIKE__WORK](./20260722.feat.menu-like.md): do **not**
replace or generalise `MenuController`’s interaction model. Keep the list shell
(`setMenu` / focus index / filter / last-action). Express calendar and chat as
**compound menu items** that can sit beside ordinary items.

Sketches under `shared/ui/menu/` (`CalendarMenu`, `ChatMenu`) stay useful as
**widget state machines**; they would be embedded in a menu item rather than
installed as the whole-menu MenuLike.

## Why consider this

- No rewrite of `MenuController` / catalog “current MenuLike” install path
- Mixes naturally with existing actions (`Today`, `Clear`, suggestions, …)
- Clicks/touches: already demo compound items with per-cell buttons
- MenuLike gaps (`getItems`, install via `setMenu`, filter/generative on the
  contract) largely evaporate for the shell

## Shape

```
menu items:
  [ .. ordinary rows .. ]
  [ calendar widget item ]   ← list focus can land on the row; inner focus owns days
  [ .. more rows .. ]
```

Chat may be even simpler: transcript as `ignored` / growing blocks + ordinary
suggestion items (no inner focus). Calendar is the spike that proves or kills
item-scoped keys.

## Hard problems (spike must answer)

1. **Item-scoped bindings** — while the calendar item has list focus *and* inner
   day focus, `$mod+j/k/h/l` (and maybe arrows) move days, not the list.
2. **Leaving the widget** — how does the user return keys to list navigation?
   Candidates: Escape / `$mod+[` pops inner focus; or “edge” of grid steps to
   next/prev menu item; or Enter activates day and exits inner mode.
3. **Catalog `when`** — default `FOCUS_NEXT_MENU_ITEM` must not fire (or must
   no-op) while inner focus is active; item bindings override or take priority.
4. **Scroll** — avoid nested scroll. Prefer chat transcript that **grows** and
   lets the menu area scroll; calendar is usually short enough.

## Non-goals for the spike

- Full MenuLike install / `setMenu({ menuLike })`
- Perfect Eliza / production calendar
- Replacing `docs/CONCEPTS.md` MenuLike section (keep both ideas until one wins)

## Spike plan

Goal: one interactive calendar **menu item** on the oneput demo page (or a tiny
AppObject) that steals list nav keys while “inside”, and releases them cleanly.

### 1. Widget without shell changes

- [ ] Reuse / slim `CalendarMenu` as day-grid state (`moveFocus` / `activate` /
      `getFocusedId`) behind a single `MenuItem` built via builder / flex children
- [ ] Pointer: day cells as buttons (existing compound-item pattern); click sets
      day + optional `onSelect`
- [ ] Render via normal `setMenu({ items: […, calendarItem, …] })` next to at
      least one ordinary item above and below

### 2. Inner focus mode

- [ ] Define two levels: **list focus on the calendar row** vs **inner focus on
      a day**
- [ ] Enter (or click on a day) enters inner mode; visual: day highlight distinct
      from list focus if needed
- [ ] Document the chosen enter/exit gestures in this ticket’s Changes

### 3. Item-scoped bindings

- [ ] Spike mechanism (pick the smallest that works):
  - **A.** AppObject `actions` with `when` that checks “inner calendar active”,
    or
  - **B.** Menu item registers/unregisters bindings on enter/exit inner mode
    via `KeysController`, or
  - **C.** Temporary override map while inner focus is set
- [ ] While inner: `$mod+h/j/k/l` (and/or arrows) call calendar `moveFocus`
- [ ] While inner: default list `FOCUS_NEXT/PREVIOUS` must not move the list

### 4. Exit back to list

- [ ] Implement at least one exit path (prefer Escape / soft exit first)
- [ ] After exit, `$mod+j/k` moves to neighbouring menu items again
- [ ] Confirm user can reach items above and below the calendar row

### 5. Chat (lighter follow-up, optional in same spike)

- [ ] Transcript as growing `ignored` rows (no inner scroll)
- [ ] Suggestions as normal focusable items — no item-scoped bindings required
- [ ] Note whether chat needs *any* of the calendar key machinery

### 6. Decision gate

After the spike, record in Changes and either:

- **Adopt** — continue implementation here; park or narrow MENU_LIKE__WORK
- **Reject** — keep MenuLike path; note why item-scoped bindings failed
- **Hybrid** — e.g. chat as items, calendar as MenuLike (unlikely; write why)

## Acceptance criteria (spike done)

- [ ] Demo: calendar row between two normal items; mouse can pick a day
- [ ] Demo: with inner focus, `$mod+j/k` (or chosen keys) move by day, not list
- [ ] Demo: exit gesture restores list `$mod+j/k` to neighbouring items
- [ ] Short write-up in Changes: binding mechanism chosen (A/B/C) + exit gesture
- [ ] Explicit adopt / reject / hybrid recommendation for MenuLike vs this path

## Relation to MenuLike

| | MenuLike ([MENU_LIKE__WORK](./20260722.feat.menu-like.md)) | This alternative |
| --- | --- | --- |
| Shell | Delegate / swap whole-area interaction | List shell unchanged |
| Calendar | Whole menu is the grid | One item is the grid |
| Mix with actions | Awkward | Natural |
| Key problem | Install + directions | Steal/release bindings |
