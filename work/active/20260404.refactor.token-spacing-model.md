# refactor: reconsider token spacing model

## Problem

Jsed TOKEN's currently carry their own trailing space by default. That simplifies some TOKEN operations, but it creates rendering problems:

- CURSOR background and marker styling includes the carried trailing space unless we compensate visually
- Native underline on `<u>` extends through the trailing space, so underlined TOKEN's look too long
- The same carried-space model makes CURSOR_STATE markers harder to place cleanly, especially for APPEND and INSERT_AFTER

The rendering issues are no longer isolated to CURSOR cosmetics. Underline behaviour suggests the current spacing model may be structurally at odds with how inline text should render.

## Current understanding

We explored three approaches in `/cursor-lab`:

1. **Pure CSS compensation** — shift the CURSOR fill left and trim it on the right with pseudo-elements. This improves the CURSOR background visually but does not fix underline semantics.
2. **Pseudo-element trailing space** — move the visible trailing space to `::after` and try `text-decoration: none`. This did not stop native underline from visually extending through the generated space.
3. **Spaces between TOKEN's** — stop storing trailing space inside TOKEN text nodes and represent spacing between adjacent TOKEN's instead. This appears to align better with both CURSOR rendering and native text decoration, but it changes the meaning of COLLAPSED_TOKEN / PADDED_TOKEN and would move spacing logic into sibling-boundary management.

The main design question is now:

- Should spacing remain a TOKEN property?
- Or should spacing become a relationship between adjacent LINE_SIBLING's?

## Candidate direction: spaces between TOKEN's

In this model:

- A TOKEN contains only visible word text
- Normal word separation is represented by text nodes between TOKEN elements
- COLLAPSED_TOKEN means "no separator after this TOKEN"
- PADDED_TOKEN becomes an explicit separator before the TOKEN (or a class expressing that intent)

This would likely fix:

- CURSOR background hugging the visible word
- Underline trailing too far through carried space
- Cleaner APPEND / INSERT_AFTER marker placement

But it would also require:

- Tokenization to emit TOKEN / separator / TOKEN sequences
- JOIN / SPLIT / INSERT operations to manage separators explicitly
- Reinterpreting COLLAPSED_TOKEN and PADDED_TOKEN as boundary semantics instead of text-content semantics

## Main plan

Work this from the `lib/` layer upward.

### 1. Change the spacing model in `lib/token.ts`

Make TOKEN text contain only visible word content. Trailing and leading spacing should no longer live inside the TOKEN text node.

Primary changes:

- `createToken` — stop appending trailing space to TOKEN text
- `getValue` — return visible TOKEN text only
- `replaceText` — replace visible TOKEN text only; stop encoding COLLAPSED / PADDED spacing in the text node
- tokenization (`replaceTextNode` and related helpers) — emit separator text nodes between adjacent TOKEN's
- `collapse` / `uncollapse` — remove/create the separator after the TOKEN rather than trimming/appending literal space in the TOKEN text
- `pad` / `unpad` — stop using leading space inside the TOKEN; instead manage a separator before the TOKEN (or boundary helper expressing the same intent)
- `remove` — update separator transfer logic when deleting TOKEN's
- `joinNext` / `joinPrevious` — merge visible TOKEN text and remove the separator boundary
- `splitBefore` / `splitAfter` — keep the structural split logic but ensure anchors / separators in the result obey the new boundary-spacing model

This likely wants explicit separator helpers in `token.ts` rather than continuing to infer spacing from TOKEN text.

### 2. Add boundary helpers where traversal needs them

`lib/sibwalk.ts` can likely stay element-based for TOKEN traversal and LINE_SIBLING walking, which is good. But TOKEN operations may need helper functions for separator text nodes around TOKEN boundaries.

Expectation:

- `getPreviousTokenSibling` / `getNextTokenSibling` probably remain mostly unchanged
- add small helpers for previous/next separator boundaries as needed by `token.ts`

`lib/taxonomy.ts` is expected to change little or not at all, as long as separators are text nodes rather than elements.

### 3. Update the upper layer after `lib/` is stable

Main files:

- `TokenCursor.ts`
- `InputManager.ts`

`TokenCursor.ts` should keep the same public concepts (APPEND, PREPEND, INSERT_AFTER, INSERT_BEFORE, TOGGLE_COLLAPSE) but operate against separator boundaries instead of TOKEN-owned spaces.

`InputManager.ts` needs review because it currently reconstructs spacing assumptions from `getValue(...)` and input whitespace handling. Once TOKEN text is spacing-free, append / insert behaviour should depend on CURSOR_STATE and separator helpers rather than TOKEN text containing carried space.

`EditManager.ts` is expected to need little semantic change beyond benefiting from `getValue` becoming more literal.

### 4. Then run exploratory scenario testing

Use `apps/jsed-demo/src/data/test_doc.html` to test the scenarios we have already been discussing:

- plain paragraphs
- INLINE_FLOW and nested inline tags
- OPAQUE_BLOCK / TRANSPARENT_BLOCK
- ISLAND adjacency and PADDED_TOKEN-like cases
- IMPLICIT_LINE cases
- underlined text and other native text decoration cases

The point is to validate the new spacing model against real document structures, not just narrow unit cases.

## Questions

- Is there any rendering-safe variant of the current carried-space model that also preserves native underline semantics?
- Does "spaces between TOKEN's" make CURSOR_STATE simpler overall, or does it merely move complexity into DOM boundary handling?
- Should this work be prototyped in the lab first, or should we write a narrow tokenizer spike against `token.ts`?

## Initial Proposal

- Stop treating trailing space as part of TOKEN text content
- Treat spacing as a boundary between adjacent LINE_SIBLING's
- Use the current lab findings as justification for a focused design pass before implementation
