# refactor: Parallel planned delete

1. Keep the existing imperative `CursorTextOps.delete()` path intact while developing the planned path. Do not rewrite the current delete operation in place. The planned version should be built in parallel, then wired through `Cursor.ts` only when parity tests are good enough.

2. Create a narrow planned-delete module under the cursor package first, not a broad mutation framework:

   ```txt
   packages/jsed/src/lib/cursor/planned-delete/
     CursorDeletePlan.ts
     CursorDeletePlanner.ts
     CursorDeleteExecutor.ts
     CursorDeleteOps.ts
     index.ts
   ```

   Start here because the experiment is specifically about `CURSOR` deletion. If the same shape works for `replace`, `insertTextAfter`, `JOIN`, and `SPLIT_BY_TOKEN`, then extract shared planning primitives later into `src/lib/mutation/`.

3. Define `CursorDeletePlan.ts` as the public data shape for the planned operation. The first version can use live DOM references only:

   ```ts
   type NodeSite = {
     parent: Node;
     previousSibling: Node | null;
     nextSibling: Node | null;
   };

   type CursorDeletePlan =
     | {
         type: 'cursor-delete-noop';
         undoable: false;
         reason: 'not-on-token';
       }
     | {
         type: 'cursor-delete';
         undoable: true;
         deletionType: 'charDeletion' | 'tokenDeletion';
         steps: CursorDeleteStep[];
         cursorBefore: HTMLElement;
         cursorAfter: HTMLElement;
         inputCursorPosition: UserInputOpts['inputCursorPosition'];
       };
   ```

   Leave room for future non-live references by keeping sites as named values rather than raw tuple positions. Do not solve operational transform now.

4. Model planned DOM changes as small step objects. Keep these specific and boring:

   ```ts
   type CursorDeleteStep =
     | { type: 'remove-token'; token: HTMLElement; site: NodeSite }
     | { type: 'remove-text'; text: Text; site: NodeSite }
     | { type: 'insert-anchor'; anchor: HTMLElement; site: NodeSite }
     | { type: 'remove-parent'; element: HTMLElement; site: NodeSite }
     | { type: 'remove-empty-tree'; removals: Array<{ element: Element; site: NodeSite }> };
   ```

   These steps should describe what will happen. They should not mutate.

5. Build `CursorDeletePlanner.ts` as the only module that reads current `CursorState` and decides the semantic delete plan. It should perform the same branching as current `delete()`:

   - if the `CURSOR` is not on a `TOKEN`, return a no-op plan
   - compute `cursorBefore`
   - ask cursor motion for previous/next cursor targets
   - plan removal of the current `TOKEN`
   - plan separator cleanup currently hidden in `token.remove`
   - choose `inputCursorPosition`
   - if there is no previous or next cursor target, plan `ANCHOR` insertion
   - if the token's visible segment becomes empty, plan parent removal and empty-tree pruning
   - compute `cursorAfter`

   Tokenization done by cursor motion remains editor maintenance, not undo-relevant mutation.

6. Do not immediately transform every existing low-level helper. Instead, create planned equivalents near the planned delete module:

   ```txt
   packages/jsed/src/lib/cursor/planned-delete/
     planTokenRemoval.ts
     planAnchorInsertion.ts
     planEmptyTreeRemoval.ts
   ```

   These helpers should mirror the semantic behavior of `token.remove`, `token.insertBefore/After/append`, `token.removeParent`, and `deleteEmptyTree`, but only compute steps. This avoids putting plan/execute pressure on the existing token and focus modules before the approach is proven.

7. Treat `token.remove()` as the main lower-level operation to transform conceptually. Its planned equivalent should inspect separators and visible siblings before mutation and emit steps for:

   - removing the `TOKEN`
   - removing paired `TRAILING_SPACE` / `LEADING_SPACE` separators when current behavior would remove them
   - returning previous/next visible siblings for the higher-level delete decision

   Keep the existing `token.remove()` in place for the imperative path.

8. Treat `deleteEmptyTree()` similarly. Its planned equivalent should walk upward without mutating and collect removable empty ancestors until the ceiling or a non-empty ancestor is reached. Because earlier planned steps may remove content, the planner may need a small "planned emptiness" helper that can answer "will this element be empty after the steps already planned?" without touching the DOM.

9. Create `CursorDeleteExecutor.ts` as a local executor for this one operation. A central executor is probably not justified yet. The planned delete work only needs a small executor that knows how to apply `CursorDeleteStep` values in order and reverse order. A central executor becomes justified later if at least two or three operations share the same step vocabulary and undo/redo logic.

10. Keep the executor deliberately dumb. It should not decide whether deletion needs an anchor, which separators to remove, or which parents are empty. It should only apply a completed plan:

    ```ts
    executeCursorDeletePlan(plan, state);
    reverseCursorDeletePlan(plan, state);
    ```

    For the first milestone, only implement forward execution. Reverse execution can be added when real undo/redo is implemented.

11. Create `CursorDeleteOps.ts` as the planned operation facade:

    ```ts
    class CursorDeleteOps {
      delete(opts?: DeleteOpts): CursorDeletePlan {
        const plan = planner.plan(opts);
        executor.execute(plan);
        if (plan.type === 'cursor-delete') {
          state.place(plan.cursorAfter, {
            inputCursorPosition: plan.inputCursorPosition
          });
        }
        return plan;
      }
    }
    ```

    This keeps `Cursor.ts` from knowing planner/executor details.

12. Wire the planned operation into `Cursor.ts` behind a single switch. The initial switch can be hardcoded false, a `CursorParams` option, or a local feature flag:

    ```ts
    delete(opts?: DeleteOpts): CursorDeleteResult {
      if (this.#state.usePlannedDelete) {
        return this.#plannedDelete.delete(opts);
      }

      return this.#ops.delete(opts);
    }
    ```

    Keep `@recordUndo` on the public `Cursor.delete()` method so both paths record whatever semantic result they return.

13. Add parity tests before making planned delete the default. For each existing `delete` case, run the imperative and planned paths against identical documents and compare:

    - final `root.innerHTML`
    - final `CURSOR` identity/value
    - undo result shape where relevant

    Start with existing cases from `CursorTextOps.test.ts`: middle `TOKEN`, last `TOKEN`, `ISLAND` no-op, `ANCHOR` no-op, token after `ISLAND`, last token after `ISLAND`, and only-token anchor insertion.

14. After parity is stable, switch `Cursor.ts` to the planned path for `delete()` and keep the old imperative path for one short cleanup window. Do not delete the old path in the same change that flips the default.

15. When real undo/redo is implemented, use the plan directly:

    - undo executes the reverse of the plan
    - redo re-executes the plan
    - local plans use live DOM references
    - future remote/OT plans can replace live references with stable ids or addresses without changing the high-level plan/executor split

16. Only extract a central mutation executor after the planned path has been repeated for more than one operation. The signal to extract is duplicated step execution across planned delete plus another operation, not the desire for an abstract framework. Until then, a local `CursorDeleteExecutor` is the lower-risk deep module.
