# discussion: dynamic `when` clause for action bindings in AppObjects

Consider allowing multiple AppObject actions to declare identical key bindings, with a dynamic `when` condition deciding which action should trigger in the current controller state.

## Current example

Today this kind of context-sensitive behavior is handled inside a single action:

```js
actions = {
  RIGHT: {
    action: () => {
      if (this.editManager.getMode() === 'editing') {
        this.editManager.cursor?.moveNext();
        return;
      }
      this.editManager.nav.REC_NEXT();
    },
    binding: {
      bindings: ['$mod+l', 'ArrowRight'],
      description: 'Move to next token or element',
      when: { menuOpen: false }
    }
  }
}
```

The question is whether Oneput should support splitting that into separate actions with the same `bindings`, where `when` is evaluated dynamically:

```js
actions = {
  REC_NEXT: {
    action: () => {
      this.editManager.cursor?.moveNext();
    },
    binding: {
      bindings: ['$mod+l', 'ArrowRight'],
      description: 'Move FOCUS to next token or element',
      when: (ctl: Controller) => {
      }
    }
  },
  MOVE_NEXT: {
    action: () => {
      this.editManager.nav.REC_NEXT();
    },
    binding: {
      bindings: ['$mod+l', 'ArrowRight'],
      description: 'Move CURSOR to next token or element',
      when: (ctl: Controller) => {
      }
    }
  }
}
```

## Questions

- What should happen if multiple dynamic `when` clauses overlap for the same binding?
- Should Oneput pick one binding and ignore the others, or surface a warning?
- If it warns, should that be `console.warn`, a Oneput notification, or both?
- When is a function-valued `when` evaluated?
- How should this appear in `BindingsEditor`, especially if the same key sequence is reused across actions?

## Current lean

This feels like a discussion item rather than a ready backlog item because it affects binding resolution semantics, conflict handling, and the editing model for binding configuration. It likely needs an explicit policy before implementation work is scoped.

## Notes from discussion

### Current dispatch behaviour

Today `when` is only a simple flag object, effectively `{ menuOpen?: boolean }`.

- Dispatch checks candidates in order and runs the first one whose `when` matches current state
- That means overlapping candidates are only safe if registration-time validation guarantees that at most one candidate can match for a given key
- If overlapping candidates slip through, runtime behaviour is effectively "first match wins"

### Function inside `when`

One possible extension is to allow a hybrid shape such as:

```ts
when: {
  menuOpen: false,
  predicate: (ctl: Controller) => boolean
}
```

This is likely better than replacing `when` with a bare function because it preserves the existing simple flags and adds an app-local dynamic predicate on top.

### Tradeoffs

- `menuOpen` and similar simple flags are serializable, editable, and understandable by validation code
- A function-valued predicate is not serializable, so persisted bindings cannot round-trip it
- `BindingsEditor` can still edit the serializable part, but not the predicate itself
- Conflict/overlap detection cannot reliably reason about arbitrary functions, so duplicate bindings with predicates become harder to validate statically
- Without extra policy, runtime behaviour falls back to "first matching predicate wins"

### Suggested constraints if we do this

- Keep serializable gating separate from executable gating
- Allow the function predicate only on AppObject-local action bindings, not on persisted user binding data
- `BindingsEditor` should only expose the serializable flags such as `menuOpen`
- Use a single field like `predicate` or `matches`, not ad hoc names like `cond1`
- Decide explicit behaviour for overlapping predicates:
  - either reject duplicate key + serializable-condition combinations when a predicate is involved
  - or document and accept "first match wins"
