# discussion: focus twice to edit

## Context

How should click/touch interact with FOCUS and the CURSOR? There are two broad models for how the user moves between navigating the document structure and editing text within it.

## Model A: focus twice to edit (current choice)

The user clicks around the document to move FOCUS between FOCUSABLE's — this is pure navigation, no CURSOR is opened. When they click the *same* FOCUSABLE a second time, quick-descend fires and the CURSOR enters edit mode on the first TOKEN.

**Two interaction modes:**
1. **Navigate** — click different FOCUSABLE's to move FOCUS around the document structure. Each click lands on a new element. The user can explore the document without committing to editing anything.
2. **Edit** — click the already-focused FOCUSABLE again. Quick-descend tokenizes the LINE and places the CURSOR. The user is now editing TOKEN's. Clicking away from the CURSOR_LINE exits edit mode (already implemented).

**Pros:**
- Clear separation between navigation and editing — clicking once never triggers tokenization
- Lets the user explore the structure (e.g. inspect which element they're on via ElementIndicator) before deciding to edit
- Simple mental model: single click = look, double click = edit

**Cons:**
- Adds one extra click to enter edit mode compared to direct entry
- "Click same element twice" may not feel intuitive without visual affordance (e.g. a hint after first focus)

## Model B: focus once to edit

Every click/touch immediately quick-descends and enters edit mode. Navigation between FOCUSABLE's would rely on keyboard shortcuts (REC_NEXT, SIB_NEXT, etc.) rather than click.

**Pros:**
- Faster path to editing — one click and you're in
- More familiar to users of conventional editors

**Cons:**
- Loses click-to-navigate — the user can't click around to explore structure without entering edit mode each time
- Would need a different mechanism for "I just want to look at this element"

## Open questions

- Should there be a visual hint after first focus that a second click will enter edit mode?
- Does this model hold up for touch interfaces where double-tap has OS-level meaning (zoom, select word)?
- Could Model A evolve into a hybrid where certain elements (e.g. simple `<p>`) enter edit mode on first click while structural elements (e.g. `<div>` with children) require the second click?
